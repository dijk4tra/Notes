`std::string` 的底层实现历史上确实主要可以概括为三类：**Eager Copy、COW、SSO**。现代 C++ 标准库实现基本都采用 **SSO**，而 **COW 已经基本退出主流实现**。

------

## 1. Eager Copy：立即深拷贝

**Eager Copy** 指的是每次拷贝 `std::string` 时，都直接复制一份完整的字符数据。

例如：

```cpp
std::string a = "hello";
std::string b = a;
```

在 Eager Copy 实现中，`b` 会拥有一块新的内存，里面存放 `"hello"` 的副本。

### 特点

优点是实现简单，语义清晰。每个字符串对象独占自己的字符缓冲区，因此修改一个字符串不会影响另一个字符串。

```cpp
b[0] = 'H';
// a 仍然是 "hello"
// b 变成 "Hello"
```

缺点是拷贝成本高。尤其是字符串很长，或者频繁传参、返回、赋值时，会产生大量内存分配和字符复制。

不过在现代 C++ 中，由于有 **移动语义**、**返回值优化 RVO/NRVO** 等机制，Eager Copy 的缺点已经被大幅缓解。

------

## 2. COW：Copy-On-Write，写时复制

**COW** 的思想是：拷贝字符串时不立刻复制字符数据，而是让多个 `std::string` 对象共享同一块底层缓冲区。只有当其中一个字符串要被修改时，才真正复制一份数据。

例如：

```cpp
std::string a = "hello";
std::string b = a;
```

此时 `a` 和 `b` 可能共享同一块 `"hello"` 数据。

当执行：

```cpp
b[0] = 'H';
```

`b` 才会复制一份新的缓冲区，然后修改自己的副本。

### 特点

优点是拷贝成本低。普通拷贝只需要复制指针、长度、引用计数等信息，不需要复制字符数组。对于大量只读字符串场景，性能很好。

缺点也很明显：

1. **写操作变复杂**
   每次修改前都要检查引用计数，必要时触发深拷贝。
2. **多线程成本高**
   共享缓冲区需要维护引用计数，引用计数通常需要原子操作或加锁，否则线程不安全。
3. **与现代 C++ 语义冲突**
   C++11 以后对 `std::string` 的迭代器、引用、指针有效性，以及并发访问语义提出了更严格要求，COW 实现变得很难满足标准要求。
4. **与移动语义收益重叠**
   C++11 引入移动语义后，很多原本依赖 COW 优化的场景可以通过 move 更直接地解决。

因此，现代主流标准库一般不再使用 COW 实现 `std::string`。

------

## 3. SSO：Short String Optimization，短字符串优化

**SSO** 是现代 `std::string` 最常见的实现方式。

它的核心思想是：字符串对象内部直接预留一小块空间，用来存放短字符串；只有当字符串长度超过这个内部空间时，才在堆上分配内存。

一个简化模型如下：

```cpp
class string {
    char* ptr;
    size_t size;
    size_t capacity;

    union {
        char local[16]; // 短字符串直接放这里
        char* heap;     // 长字符串使用堆内存
    };
};
```

实际实现会更复杂，但思想类似。

例如：

```cpp
std::string s = "hello";
```

`"hello"` 很短，可能直接存储在 `std::string` 对象内部，不发生堆分配。

而：

```cpp
std::string s = "this is a very long string ...";
```

长度超过 SSO 容量后，才会分配堆内存。

### 特点

SSO 的优点非常明显：

1. **短字符串无需堆分配**
   很多实际字符串都很短，比如文件名、字段名、状态码、JSON key、短日志信息等。SSO 可以显著减少 `malloc/free` 开销。
2. **缓存友好**
   短字符串数据和 `std::string` 对象本身在一起，局部性更好。
3. **没有 COW 的共享问题**
   每个字符串对象拥有自己的存储，不需要引用计数，也不会因为写操作触发隐式共享分离。
4. **适合现代 C++**
   配合移动语义、RVO 等优化，SSO 成为现代标准库的主流选择。

缺点是 `std::string` 对象本身通常会变大，因为对象内部要预留短字符串缓冲区。此外，SSO 的具体容量依赖实现，不同标准库、不同平台可能不同。

例如常见实现中，短字符串容量可能是：

```cpp
sizeof(std::string) == 24 或 32 字节
SSO 可容纳约 15 或 22 个 char
```

但这些数值不是 C++ 标准保证的。

------

## 三者对比

| 实现方式   | 拷贝成本                 | 修改成本 | 是否共享数据 | 堆分配情况       | 现代使用情况                 |
| ---------- | ------------------------ | -------- | ------------ | ---------------- | ---------------------------- |
| Eager Copy | 高                       | 低       | 否           | 拷贝通常分配     | 早期常见，现代可与 move 配合 |
| COW        | 低                       | 较高     | 是           | 写时可能分配     | 现代基本不用                 |
| SSO        | 短字符串低，长字符串正常 | 低       | 否           | 短字符串无堆分配 | 主流实现                     |

------

## 总结

可以这样理解：

**Eager Copy** 是“拷贝时立刻复制数据”，简单但可能浪费。

**COW** 是“先共享，修改时再复制”，读多写少时高效，但在现代 C++ 中有线程和语义问题。

**SSO** 是“短字符串直接放对象内部，长字符串才上堆”，兼顾性能、简单性和现代 C++ 语义，因此成为当前主流实现。
