# STL

## 容器、迭代器与算法

### 1. STL 包括哪些内容？

#### 面试回答

STL 是 C++ 标准库中泛型编程思想最核心的一部分，主要包括容器、迭代器、算法、函数对象、适配器和分配器。容器负责保存数据，算法负责通用操作，迭代器把容器和算法连接起来，使算法不依赖具体容器类型。STL 的核心思想是泛型编程、数据结构与算法解耦、通过迭代器提供统一访问接口。

#### 1. 容器 containers

容器用于存储对象，按组织方式可分为：

| 类别 | 典型容器 | 特点 |
| --- | --- | --- |
| 序列容器 | `vector`、`deque`、`list`、`forward_list`、`array` | 元素按线性顺序组织 |
| 有序关联容器 | `set`、`map`、`multiset`、`multimap` | 通常基于红黑树，按 key 有序 |
| 无序关联容器 | `unordered_set`、`unordered_map` | 基于哈希表，平均 O(1) 查找 |
| 容器适配器 | `stack`、`queue`、`priority_queue` | 对底层容器接口做限制或包装 |

#### 2. 迭代器 iterators

迭代器是容器和算法之间的桥梁。算法只需要通过迭代器访问区间，不需要知道底层容器是数组、链表还是树。

```cpp
std::vector<int> v{1, 2, 3};
std::sort(v.begin(), v.end());
```

迭代器类别包括输入迭代器、输出迭代器、前向迭代器、双向迭代器、随机访问迭代器、连续迭代器等。不同算法对迭代器能力有要求，例如 `std::sort` 需要随机访问迭代器，所以不能直接用于 `std::list`。

#### 3. 算法 algorithms

算法包括查找、排序、计数、拷贝、删除、变换、堆操作、集合操作等。常见算法：

- `find/find_if`
- `sort/stable_sort`
- `lower_bound/upper_bound`
- `copy/remove/remove_if`
- `for_each/transform`
- `make_heap/push_heap/pop_heap`

#### 4. 函数对象 function objects

函数对象是重载了 `operator()` 的对象，可像函数一样调用，也可以保存状态。

```cpp
struct Greater {
    bool operator()(int a, int b) const {
        return a > b;
    }
};
```

lambda 本质上也是函数对象，常用于 STL 算法的谓词或比较器。

#### 5. 适配器 adapters

适配器是在已有组件上包装出另一种接口。

- 容器适配器：`stack`、`queue`、`priority_queue`。
- 迭代器适配器：`reverse_iterator`、`back_insert_iterator`。
- 函数适配器：`std::bind` 等。

#### 6. 分配器 allocators

分配器负责容器的内存分配和释放。默认是 `std::allocator<T>`。普通业务代码很少直接自定义 allocator，但高性能服务、内存池、共享内存容器等场景可能会用到。

#### 常见追问

- **STL 的核心优势是什么？**  
  泛型、复用、类型安全、算法和容器解耦、复杂度有标准保证。

- **STL 是否只包含容器？**  
  不是。容器只是 STL 的一部分，算法、迭代器、适配器、函数对象同样重要。

### 2. `vector` 底层实现。

#### 面试回答

`vector` 是动态连续数组，底层使用一段连续内存保存元素，支持 O(1) 随机访问和尾部插入均摊 O(1)。典型实现会维护三个指针或等价成员：起始位置、当前元素末尾、容量末尾。当容量不足时，`vector` 会分配更大的连续空间，把旧元素移动或拷贝过去，再释放旧空间。

#### 1. 典型内部结构

可以粗略理解为：

```cpp
template <class T>
class vector {
    T* begin_;
    T* end_;
    T* cap_;
};
```

含义：

- `begin_` 指向第一个元素。
- `end_` 指向最后一个元素的下一个位置。
- `cap_` 指向已分配内存末尾的下一个位置。

```text
begin_              end_                  cap_
  |                  |                     |
  v                  v                     v
+----+----+----+----+---------------------+
| e0 | e1 | e2 | e3 | uninitialized space |
+----+----+----+----+---------------------+
```

`size() = end_ - begin_`，`capacity() = cap_ - begin_`。

#### 2. 扩容过程

当 `push_back/emplace_back` 时容量不够，扩容大致流程：

1. 申请一块更大的连续内存。
2. 将旧元素移动或拷贝到新内存。
3. 在新位置构造新元素。
4. 析构旧内存中的元素。
5. 释放旧内存。
6. 更新内部指针。

```cpp
std::vector<std::string> v;
v.reserve(100);       // 提前分配容量，减少扩容次数
v.emplace_back("hi"); // 直接在尾部构造
```

#### 3. 复杂度

| 操作 | 复杂度 |
| --- | --- |
| `operator[]` / `at` | O(1) |
| 尾部插入 | 均摊 O(1) |
| 中间插入/删除 | O(n) |
| 查找指定值 | O(n) |
| 扩容 | O(n) |

#### 4. 优缺点

优点：

- 连续内存，缓存局部性好。
- 支持随机访问。
- 可与 C API 通过 `data()` 交互。
- 实际工程中遍历性能通常很好。

缺点：

- 中间插入删除需要移动元素。
- 扩容会导致迭代器、指针、引用失效。
- 要求有一段连续内存，大容量时分配可能失败。

#### 常见追问

- **`size` 和 `capacity` 区别？**  
  `size` 是已经构造的元素个数；`capacity` 是当前已分配空间最多能容纳多少元素。

- **`reserve` 和 `resize` 区别？**  
  `reserve` 只改变容量，不构造元素；`resize` 改变元素个数，会构造或析构元素。

### 3. `vector` 和 `deque` 的区别？

#### 面试回答

`vector` 是单段连续内存，随机访问快，尾插均摊 O(1)，但头部和中间插入删除代价高，扩容会整体搬迁。`deque` 是双端队列，底层通常由多段固定大小缓冲区和中控表组成，支持头尾 O(1) 插入删除，随机访问也是 O(1)，但内存不是整体连续，缓存局部性通常不如 `vector`。需要连续内存或与 C API 交互时选 `vector`，需要频繁头尾插删时可选 `deque`。

#### 1. 存储结构区别

`vector`：

```text
+----+----+----+----+----+
| e0 | e1 | e2 | e3 | e4 |
+----+----+----+----+----+
一整段连续内存
```

`deque`：

```text
map:
+-----+-----+-----+
| p0  | p1  | p2  |
+-----+-----+-----+
  |     |     |
 block block block
```

`deque` 的元素分布在多个缓冲区中，中控表保存每个缓冲区地址。随机访问时通过下标计算所在块和块内偏移。

#### 2. 操作复杂度对比

| 操作 | `vector` | `deque` |
| --- | --- | --- |
| 随机访问 | O(1)，最快 | O(1)，略慢 |
| 尾插 | 均摊 O(1) | O(1) |
| 头插 | O(n) | O(1) |
| 中间插入删除 | O(n) | O(n) |
| 内存连续性 | 整体连续 | 分段连续 |
| 扩容影响 | 可能整体搬迁 | 通常不整体搬迁 |

#### 3. 迭代器失效差异

`vector` 重新分配会使所有迭代器、指针、引用失效。`deque` 插入删除头尾时规则更复杂，可能使迭代器失效，但对已有元素引用的影响和具体操作有关。面试中可概括：`deque` 不保证整体连续，迭代器实现也比 `vector` 复杂。

#### 4. 选择建议

- 大量遍历、随机访问、需要连续内存：`vector`。
- 频繁头尾插入删除：`deque`。
- 中间频繁插入删除且已定位节点：`list`。

> [!CAUTION]
> `deque` 名字是 double-ended queue，不是 queue 的简单增强版。它支持随机访问，但不保证内存整体连续。

### 4. `vector` 和 `list` 的本质区别？

#### 面试回答

`vector` 的本质是连续动态数组，`list` 的本质是双向链表。`vector` 支持 O(1) 随机访问，遍历缓存友好，尾插效率高；但中间插入删除要移动元素，扩容会导致迭代器失效。`list` 在已知节点位置时插入删除 O(1)，迭代器稳定性好；但不支持随机访问，查找必须 O(n) 遍历，每个节点还有额外指针开销，缓存局部性差。

#### 1. 内存结构

`vector`：

```text
连续内存: [e0][e1][e2][e3]
```

`list`：

```text
node <-> node <-> node <-> node
每个节点独立分配，包含 prev/next/data
```

#### 2. 复杂度对比

| 操作 | `vector` | `list` |
| --- | --- | --- |
| 随机访问 `a[i]` | O(1) | 不支持，需 O(n) |
| 顺序遍历 | 快，缓存友好 | 慢，指针跳转多 |
| 尾部插入 | 均摊 O(1) | O(1) |
| 头部插入 | O(n) | O(1) |
| 已知位置插入/删除 | O(n)，移动元素 | O(1)，改指针 |
| 查找某值 | O(n) | O(n) |
| 内存开销 | 较低 | 每节点额外两个指针 |

#### 3. 为什么实际工程中 `vector` 常常更快？

虽然 `list` 的插入删除复杂度看起来更优，但前提是已经拿到插入/删除位置的迭代器。实际场景中经常需要先查找位置，而查找是 O(n)。同时 `vector` 连续内存带来很好的缓存局部性，CPU 预取友好；`list` 节点分散，容易 cache miss。

#### 4. 选择建议

- 默认优先考虑 `vector`。
- 需要频繁中间插删，并且已经持有迭代器，且不需要随机访问，再考虑 `list`。
- 如果只需要头尾操作，考虑 `deque`。

### 5. `vector`、`list` 在添加删除的效率方面有什么不同？

#### 面试回答

`vector` 在尾部添加元素均摊 O(1)，但头部或中间添加删除通常是 O(n)，因为要移动后续元素；如果触发扩容，还要整体移动或拷贝所有元素。`list` 在已知位置时添加删除是 O(1)，只需要修改节点指针，但找到位置通常要 O(n)。所以不能简单说 `list` 插删一定比 `vector` 快，要结合定位成本、缓存局部性和内存分配成本。

#### `vector` 添加删除

尾插：

```cpp
std::vector<int> v;
v.push_back(1); // 容量足够时 O(1)
```

中间插入：

```cpp
v.insert(v.begin() + 1, 100); // 后续元素后移，O(n)
```

删除：

```cpp
v.erase(v.begin() + 1); // 后续元素前移，O(n)
```

特点：

- 尾插高效。
- 中间插删要移动元素。
- 扩容时代价大且迭代器失效。
- 连续内存带来高遍历性能。

#### `list` 添加删除

```cpp
std::list<int> lst{1, 2, 3};
auto it = std::next(lst.begin());
lst.insert(it, 100); // 已有 it 时 O(1)
lst.erase(it);       // 删除 it 指向节点 O(1)
```

特点：

- 已定位位置后插入删除 O(1)。
- 插入删除不会移动其他元素。
- 其他节点迭代器通常不失效。
- 查找位置和随机访问慢。
- 每次插入可能有节点动态分配成本。

#### 常见追问

- **大量删除 `vector` 中满足条件的元素怎么写？**  
  可以使用 erase-remove 惯用法。

```cpp
v.erase(std::remove_if(v.begin(), v.end(), pred), v.end());
```

- **为什么 `list` 不支持 `operator[]`？**  
  链表节点不连续，无法通过首地址加偏移直接定位第 n 个元素。

### 6. 释放 `vector` 内存的处理方式。

#### 面试回答

`vector::clear()` 只销毁元素，让 `size()` 变为 0，通常不会释放已分配容量。若要请求释放多余容量，可使用 `shrink_to_fit()`，但标准不强制保证一定释放。若希望明确释放容量，常用临时空 vector 与当前 vector 交换，即 `std::vector<T>().swap(v)`。如果 vector 很快还要继续插入元素，保留容量反而更高效。

#### 1. `clear`

```cpp
std::vector<int> v(10000);
v.clear();
```

效果：

- `size()` 变为 0。
- 元素被销毁。
- `capacity()` 通常不变。
- 后续再次插入可复用已有内存。

#### 2. `shrink_to_fit`

```cpp
v.shrink_to_fit();
```

效果：

- 向实现请求把容量缩小到接近 `size()`。
- 是非强制请求，标准不保证一定释放。
- 可能导致重新分配，从而使迭代器、指针、引用失效。

#### 3. swap 技巧

释放全部空间：

```cpp
std::vector<int>().swap(v);
```

保留当前元素但收缩容量：

```cpp
std::vector<int>(v).swap(v);
```

不过现代 C++ 中优先考虑 `shrink_to_fit()`，只有确实需要强控制时再使用 swap 技巧。

#### 4. 对象销毁

当 `vector` 对象生命周期结束时，其析构函数会自动销毁所有元素并释放内存。

```cpp
{
    std::vector<int> tmp(10000);
} // tmp 析构，内存释放
```

#### 常见追问

- **`clear` 后迭代器还能用吗？**  
  指向元素的迭代器失效，因为元素已经被销毁。

- **为什么 `clear` 不释放容量？**  
  为了避免频繁释放和重新分配，提高后续插入性能。

### 7. `vector` 迭代器失效的情况有哪些？

#### 面试回答

`vector` 的迭代器、指针和引用失效主要来自两类操作：重新分配内存和移动元素。只要操作导致扩容，所有迭代器、指针、引用都会失效；如果没有扩容，插入会使插入位置及其后的迭代器失效，删除会使删除位置及其后的迭代器失效。尾插未扩容时，通常只有 `end()` 失效。

#### 失效规则

| 操作 | 是否触发扩容 | 失效情况 |
| --- | --- | --- |
| `push_back/emplace_back` | 是 | 全部迭代器、指针、引用失效 |
| `push_back/emplace_back` | 否 | `end()` 失效，已有元素引用通常仍有效 |
| `insert/emplace` | 是 | 全部失效 |
| `insert/emplace` | 否 | 插入位置及其后的迭代器/引用失效 |
| `erase` | 不涉及扩容 | 删除位置及其后的迭代器/引用失效 |
| `clear` | 不释放容量 | 所有元素迭代器失效 |
| `reserve` | 新容量大于旧容量 | 全部失效 |
| `shrink_to_fit` | 若发生重分配 | 全部失效 |

#### 正确删除写法

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (*it % 2 == 0) {
        it = v.erase(it);
    } else {
        ++it;
    }
}
```

错误写法：

```cpp
for (auto it = v.begin(); it != v.end(); ++it) {
    if (*it % 2 == 0) {
        v.erase(it); // it 已失效，循环里 ++it 又使用失效迭代器
    }
}
```

#### 常见追问

- **为什么删除位置之后的迭代器会失效？**  
  因为后面的元素会整体前移，原迭代器指向的对象位置和语义都改变了。

- **如何减少扩容导致的失效？**  
  提前 `reserve` 合理容量，减少重新分配次数。

### 8. `map` 和 `unordered_map` 的区别。

#### 面试回答

`map` 是有序关联容器，通常基于红黑树实现，元素按 key 排序，查找、插入、删除复杂度为 O(log n)，支持有序遍历和范围查询。`unordered_map` 是无序关联容器，通常基于哈希表实现，平均查找、插入、删除 O(1)，但不保证遍历顺序，最坏情况下可能退化到 O(n)。需要顺序、范围查询、稳定复杂度时选 `map`；追求平均查找性能且不关心顺序时选 `unordered_map`。

#### 对比表

| 对比项 | `map` | `unordered_map` |
| --- | --- | --- |
| 底层结构 | 红黑树 | 哈希表 |
| 元素顺序 | 按 key 有序 | 无序 |
| 查找复杂度 | O(log n) | 平均 O(1)，最坏 O(n) |
| 范围查询 | 支持 | 不适合 |
| 迭代器遍历 | 有序 | 桶顺序，非稳定排序 |
| key 要求 | 可比较，默认 `operator<` | 可哈希且可相等比较 |
| 内存 | 节点指针开销 | 桶数组 + 节点/槽位开销 |

#### 示例

```cpp
std::map<int, std::string> ordered;
ordered[2] = "b";
ordered[1] = "a";
// 遍历顺序：1, 2

std::unordered_map<int, std::string> hashed;
hashed[2] = "b";
hashed[1] = "a";
// 遍历顺序不保证
```

#### 选择建议

用 `map`：

- 需要按 key 有序输出。
- 需要 `lower_bound/upper_bound`。
- 需要范围查询。
- 希望最坏复杂度更稳定。

用 `unordered_map`：

- 只需要快速按 key 查找。
- 不关心遍历顺序。
- key 的哈希函数质量可靠。
- 数据规模较大且查询频繁。

#### 常见追问

- **为什么 `unordered_map` 最坏 O(n)？**  
  哈希冲突严重时，大量 key 落入同一桶，查找需要遍历桶内元素。

- **`map` 的迭代器插入后会失效吗？**  
  插入通常不使已有迭代器失效；删除只使被删除元素的迭代器失效。

### 9. STL 中 `unordered_map` 和 `unordered_set` 有什么区别？

#### 面试回答

`unordered_map` 和 `unordered_set` 底层通常都是哈希表，区别在于存储内容和接口语义。`unordered_map` 存储 key-value 键值对，通过 key 查找 value；`unordered_set` 只存 key，用于快速判断元素是否存在或去重。二者都不保证元素有序，平均查找、插入、删除复杂度为 O(1)。

#### 数据模型

```cpp
std::unordered_set<std::string> words;
words.insert("cpp");

std::unordered_map<std::string, int> freq;
freq["cpp"]++;
```

`unordered_set<T>` 中，元素本身就是 key。`unordered_map<K, V>` 中，节点通常保存 `std::pair<const K, V>`。

#### 对比表

| 对比项 | `unordered_set` | `unordered_map` |
| --- | --- | --- |
| 保存内容 | key | key-value |
| 主要用途 | 去重、存在性判断 | 映射、计数、缓存 |
| 查找返回 | 迭代器指向 key | 迭代器指向 pair |
| 访问 value | 无 | `it->second` |
| 示例 | 黑名单集合 | 单词频率统计 |

#### 共同点

- 都依赖哈希函数 `std::hash<Key>`。
- 都依赖相等比较 `std::equal_to<Key>`。
- 都有桶 bucket 和负载因子 load factor。
- rehash 会导致迭代器失效。
- 遍历顺序不稳定，不应依赖。

#### 常见追问

- **为什么 unordered 容器中 key 不能随便修改？**  
  因为 key 决定哈希值和桶位置，修改 key 会破坏哈希表结构。

- **如何自定义 key？**  
  需要提供哈希函数和相等比较函数。

### 10. 自己实现 `unordered_map` 会考虑什么问题？

#### 面试回答

实现 `unordered_map` 需要考虑哈希函数、桶数组、冲突解决、负载因子、扩容 rehash、插入删除查找流程、迭代器、异常安全、内存管理、key 相等比较、并发安全和哈希攻击等问题。常见实现方式是链地址法：桶数组中每个桶挂一个链表或节点集合；也可以用开放寻址法，通过探测寻找空槽。

#### 1. 基本结构

链地址法：

```text
buckets:
0 -> node -> node
1 -> nullptr
2 -> node
3 -> node -> node -> node
```

节点保存：

```cpp
template <class K, class V>
struct Node {
    K key;
    V value;
    Node* next;
};
```

#### 2. 核心操作

查找：

1. 计算 `hash(key)`。
2. 通过 `hash % bucket_count` 定位桶。
3. 遍历桶内节点，用相等比较找到 key。

插入：

1. 先查找 key 是否存在。
2. 不存在则插入节点。
3. 更新元素数量。
4. 如果负载因子过高，触发 rehash。

删除：

1. 定位桶。
2. 在桶内链表中找到节点。
3. 修改前驱指针并释放节点。

#### 3. 负载因子和 rehash

负载因子：

```text
load_factor = size / bucket_count
```

负载因子过高，冲突增多，查找变慢。因此需要扩容桶数组并重新计算每个元素所在桶。

```cpp
if (size_ > max_load_factor_ * buckets_.size()) {
    rehash(buckets_.size() * 2);
}
```

#### 4. 冲突解决方式

| 方式 | 思路 | 优点 | 缺点 |
| --- | --- | --- | --- |
| 链地址法 | 桶内挂链表/节点 | 删除简单，负载因子可超过 1 | 节点分散，缓存较差 |
| 开放寻址 | 冲突后继续探测空槽 | 缓存友好 | 删除复杂，负载因子不能太高 |
| 线性探测 | 顺序找下一个槽 | 简单、局部性好 | 聚集问题 |
| 二次探测 | 按平方偏移探测 | 减轻聚集 | 实现稍复杂 |
| 双重哈希 | 用第二哈希决定步长 | 分布更好 | 哈希成本高 |

#### 5. 工程问题

- 哈希函数质量影响性能。
- rehash 会造成性能抖动。
- 迭代器在 rehash 后失效。
- 插入时要考虑异常安全。
- 是否允许多线程并发访问，若允许要分段锁或读写锁。
- 恶意输入可能制造大量冲突，需要防哈希攻击。

#### 常见追问

- **为什么要有 `max_load_factor`？**  
  控制平均桶长度，平衡空间和时间。

- **开放寻址删除为什么麻烦？**  
  删除后不能简单置空，否则会中断后续探测链，通常需要 tombstone 标记。

### 11. `clear` 和 `erase` 的区别？

#### 面试回答

`clear` 用于清空整个容器，删除所有元素，使 `size()` 变为 0；`erase` 用于删除指定位置、指定 key 或指定区间的元素。`clear` 是整体操作，`erase` 是局部操作。对于 `vector`，`clear` 通常不释放容量，`erase` 会移动后续元素；对于关联容器，`erase(key)` 返回删除元素数量。

#### 1. `clear`

```cpp
std::vector<int> v{1, 2, 3};
v.clear();
```

效果：

- 删除所有元素。
- `size()` 变为 0。
- 元素析构。
- 对 `vector/string` 等，容量通常保留。
- 所有指向元素的迭代器失效。

#### 2. `erase`

按迭代器删除：

```cpp
auto it = v.erase(v.begin()); // 返回下一个有效位置
```

按区间删除：

```cpp
v.erase(v.begin(), v.begin() + 2);
```

关联容器按 key 删除：

```cpp
std::map<int, std::string> m;
size_t n = m.erase(10); // 删除 key 为 10 的元素，返回删除数量
```

#### 3. 不同容器中的差异

| 容器 | `clear` | `erase` |
| --- | --- | --- |
| `vector` | 清空元素，容量通常保留 | 删除后移动后续元素 |
| `list` | 删除所有节点 | 删除节点，其他迭代器通常不失效 |
| `map/set` | 删除所有节点 | 删除指定 key/迭代器/区间 |
| `unordered_map` | 清空元素，桶数量可能保留 | 删除指定 key/节点 |

#### 常见追问

- **`clear` 会不会释放内存？**  
  对多数容器会释放节点或析构元素；但 `vector` 的容量通常不会释放。

- **边遍历边 erase 怎么写？**  
  使用 `erase` 返回值更新迭代器。

### 12. 迭代器失效问题。

#### 面试回答

迭代器失效是指容器经过插入、删除、扩容、rehash 等操作后，原来的迭代器不再能安全使用。失效规则取决于容器底层结构：连续容器因为元素移动和重新分配更容易失效；链表容器插入删除通常只影响被删节点；树形关联容器插入通常不影响已有迭代器，删除只影响被删元素；无序容器 rehash 会使迭代器失效。

#### 1. 序列容器

`vector`：

- 扩容：全部迭代器、指针、引用失效。
- 插入未扩容：插入位置及其后失效。
- 删除：删除位置及其后失效。

`deque`：

- 头尾插删规则比 `vector` 复杂，迭代器容易失效。
- 不应长期保存 `deque` 迭代器后再做结构修改。

`list/forward_list`：

- 插入不影响已有迭代器。
- 删除只使被删除元素的迭代器失效。

#### 2. 有序关联容器

`map/set/multimap/multiset` 通常基于节点：

- 插入不使已有迭代器失效。
- 删除只使被删除元素的迭代器失效。
- 其他节点地址稳定。

#### 3. 无序关联容器

`unordered_map/unordered_set`：

- 插入如果触发 rehash，所有迭代器失效。
- 插入不触发 rehash，通常已有迭代器仍有效。
- 删除只使被删元素迭代器失效。
- rehash 后元素分桶改变，遍历顺序也可能改变。

#### 4. 安全删除模式

```cpp
for (auto it = c.begin(); it != c.end(); ) {
    if (shouldErase(*it)) {
        it = c.erase(it);
    } else {
        ++it;
    }
}
```

> [!TIP]
> 遍历删除时优先写 `it = c.erase(it)`，不要删除后再 `++it`。

#### 常见追问

- **引用和指针也会失效吗？**  
  会。很多规则同时适用于迭代器、指针和引用，特别是 `vector` 扩容。

- **如何减少失效问题？**  
  提前 `reserve`、避免在遍历中修改结构、使用返回迭代器、选择迭代器稳定的容器。

### 13. `swap` 函数的作用？

#### 面试回答

`swap` 用于交换两个对象的内容。对于 STL 容器，`swap` 通常只是交换内部指针、大小、容量、比较器、分配器等状态，而不是逐个交换元素，因此效率很高。它常用于快速交换两个容器、释放 vector 容量、实现 copy-and-swap 赋值、提高异常安全。自定义类型也应该提供高效且尽量 `noexcept` 的 `swap`。

#### 1. 容器 swap

```cpp
std::vector<int> a{1, 2, 3};
std::vector<int> b{4, 5};
a.swap(b);
```

交换后：

- `a` 拥有原来 `b` 的元素。
- `b` 拥有原来 `a` 的元素。
- 通常不逐个移动元素。

#### 2. 释放 vector 内存

```cpp
std::vector<int>().swap(v);
```

临时空 vector 与 `v` 交换后，`v` 变为空；临时对象析构时释放原来 `v` 的内存。

#### 3. copy-and-swap

```cpp
class Buffer {
public:
    Buffer& operator=(Buffer other) {
        swap(other);
        return *this;
    }

    void swap(Buffer& other) noexcept {
        std::swap(data_, other.data_);
        std::swap(size_, other.size_);
    }

private:
    char* data_{};
    size_t size_{};
};
```

这种写法利用参数副本和 swap 实现赋值，异常安全性好。

#### 4. ADL 与自定义 swap

泛型代码中常写：

```cpp
using std::swap;
swap(a, b);
```

这样既能使用标准 `std::swap`，也能通过 ADL 找到用户自定义的高效 swap。

#### 常见追问

- **`swap` 会不会导致迭代器失效？**  
  容器 swap 后，迭代器通常仍指向原来的元素，但这些元素现在属于另一个容器。具体还要看容器和分配器规则。

- **为什么 swap 最好是 `noexcept`？**  
  标准库在移动、排序、容器操作中会根据 `noexcept` 选择更高效或更安全的路径。

### 14. 简述 STL 容器相关知识、特性等？

#### 面试回答

STL 容器按数据组织方式可分为序列容器、有序关联容器、无序关联容器和容器适配器。选择容器时要考虑访问方式、插入删除位置、是否需要有序、是否需要随机访问、内存局部性、迭代器稳定性和复杂度要求。一般工程中默认优先考虑 `vector`，需要有序映射用 `map`，需要快速哈希查找用 `unordered_map`，需要头尾操作用 `deque`，需要稳定节点迭代器和频繁中间插删时考虑 `list`。

#### 容器分类

| 类别 | 容器 | 底层结构 | 典型特点 |
| --- | --- | --- | --- |
| 序列容器 | `array` | 固定数组 | 固定大小，连续内存 |
| 序列容器 | `vector` | 动态数组 | 连续内存，随机访问快 |
| 序列容器 | `deque` | 分段数组 | 头尾插删快 |
| 序列容器 | `list` | 双向链表 | 节点稳定，中间插删快 |
| 序列容器 | `forward_list` | 单向链表 | 更省空间，只能前向 |
| 有序关联 | `map/set` | 红黑树 | 按 key 有序，O(log n) |
| 无序关联 | `unordered_map/set` | 哈希表 | 平均 O(1)，无序 |
| 适配器 | `stack/queue` | 默认 deque | 限制接口 |
| 适配器 | `priority_queue` | 堆 | 取最大/最小优先 |

#### 复杂度速记

| 容器 | 随机访问 | 查找 | 插入删除 |
| --- | --- | --- | --- |
| `vector` | O(1) | O(n) | 尾部均摊 O(1)，中间 O(n) |
| `deque` | O(1) | O(n) | 头尾 O(1)，中间 O(n) |
| `list` | 不支持 | O(n) | 已知位置 O(1) |
| `map/set` | 不支持 | O(log n) | O(log n) |
| `unordered_map/set` | 不支持 | 平均 O(1) | 平均 O(1) |

#### 选择建议

- 需要连续内存、随机访问、频繁遍历：`vector`。
- 固定大小数组：`array`。
- 频繁头尾操作：`deque`。
- 频繁中间插删且已定位：`list`。
- 需要有序和范围查询：`map/set`。
- 只需要快速查找：`unordered_map/unordered_set`。
- 只暴露栈/队列语义：`stack/queue/priority_queue`。

#### 常见追问

- **为什么 `vector` 是默认首选？**  
  内存连续、缓存友好、实现简单、遍历快，多数场景实际性能好。

- **容器适配器为什么叫适配器？**  
  因为它们不是全新底层结构，而是基于已有容器限制接口形成新的抽象。

### 15. STL 中 `vector`、`list`、`map` 在内存中的数据结构有什么区别？

#### 面试回答

`vector` 在内存中是一段连续动态数组，元素紧密排列；`list` 是由离散节点组成的双向链表，每个节点包含数据和前后指针；`map` 通常是红黑树，每个节点包含 key-value、左右孩子指针、父指针和颜色信息。连续结构缓存友好但插删会移动元素，节点结构迭代器更稳定但内存开销大、缓存局部性差。

#### 1. `vector` 内存结构

```text
+----+----+----+----+
| e0 | e1 | e2 | e3 |
+----+----+----+----+
```

特点：

- 元素连续。
- 可用 `data()` 获取底层数组指针。
- 随机访问 O(1)。
- 扩容需要整体搬迁。

#### 2. `list` 内存结构

```text
+------+      +------+      +------+
|prev  | <--> |prev  | <--> |prev  |
|data  |      |data  |      |data  |
|next  |      |next  |      |next  |
+------+      +------+      +------+
```

特点：

- 每个节点单独分配。
- 插入删除节点只改指针。
- 不支持随机访问。
- 每个节点至少额外两个指针。

#### 3. `map` 内存结构

典型红黑树节点：

```text
        [key=5, black]
        /           \
 [key=2, red]   [key=8, red]
```

节点通常包含：

- key 或 key-value。
- 左孩子指针。
- 右孩子指针。
- 父指针。
- 颜色位。

特点：

- 按 key 有序。
- 查找、插入、删除 O(log n)。
- 节点地址相对稳定。
- 内存开销比 `vector` 大。

#### 对比表

| 容器 | 内存连续 | 节点额外开销 | 随机访问 | 迭代器稳定性 | 缓存友好 |
| --- | --- | --- | --- | --- | --- |
| `vector` | 是 | 低 | 支持 | 扩容/插删易失效 | 好 |
| `list` | 否 | 高 | 不支持 | 较好 | 差 |
| `map` | 否 | 高 | 不支持 | 插入稳定，删除当前失效 | 一般/较差 |

### 16. `erase` 的返回值。

#### 面试回答

`erase` 的返回值取决于容器和重载形式。对序列容器来说，`erase(iterator)` 返回被删除元素后面的迭代器，`erase(first, last)` 返回删除区间后面的迭代器。对关联容器来说，C++11 起 `erase(iterator)` 也返回后继迭代器，而 `erase(key)` 返回删除的元素数量。利用返回值可以安全地边遍历边删除。

#### 1. 序列容器

```cpp
std::vector<int> v{1, 2, 3, 4};
auto it = v.erase(v.begin() + 1); // 删除 2，it 指向 3
```

区间删除：

```cpp
auto next = v.erase(v.begin(), v.begin() + 2);
```

返回删除区间之后的位置。如果删除到末尾，返回 `end()`。

#### 2. 关联容器

```cpp
std::map<int, std::string> m;
auto it = m.find(10);
if (it != m.end()) {
    it = m.erase(it); // C++11 起返回后继迭代器
}

size_t count = m.erase(20); // 按 key 删除，返回删除数量
```

对 `map/set`，key 唯一，`erase(key)` 返回 0 或 1；对 `multimap/multiset`，可能返回大于 1。

#### 3. 安全遍历删除

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (*it % 2 == 0) {
        it = v.erase(it);
    } else {
        ++it;
    }
}
```

对 `map`：

```cpp
for (auto it = m.begin(); it != m.end(); ) {
    if (it->first < 0) {
        it = m.erase(it);
    } else {
        ++it;
    }
}
```

#### 常见追问

- **为什么 erase 后不能继续用原来的迭代器？**  
  因为它指向的元素已经被删除，迭代器失效。

- **C++03 的关联容器 `erase(iterator)` 返回什么？**  
  旧标准中返回 `void`，常见写法是 `m.erase(it++)`。现代 C++ 推荐用返回值。
