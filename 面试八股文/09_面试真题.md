# 面试真题

### 1. 头文件头部为什么使用 `#ifndef/#define/#endif`？

#### 面试回答

这是头文件保护宏，也叫 include guard，作用是防止同一个头文件在同一个翻译单元中被重复包含。C/C++ 的 `#include` 本质是预处理阶段的文本展开，如果一个头文件被多次展开，其中的类型定义、全局变量定义、函数定义等可能造成重复定义或编译错误。使用 `#ifndef/#define/#endif` 后，第一次包含会定义宏，后续再次包含时宏已存在，预处理器会跳过文件主体。

#### 典型写法

```cpp
#ifndef C_CPP_BASIC_TEST_H
#define C_CPP_BASIC_TEST_H

// declarations
class Test {};

#endif // C_CPP_BASIC_TEST_H
```

#### 为什么需要它

假设 `a.cpp` 同时包含：

```cpp
#include "x.h"
#include "y.h"
```

而 `x.h` 和 `y.h` 都包含 `common.h`。如果 `common.h` 中定义了结构体：

```cpp
struct Config {
    int port;
};
```

没有保护宏时，`common.h` 会被展开两次，同一个翻译单元中出现两次 `struct Config` 定义，导致重定义错误。

#### `#pragma once`

现代编译器通常支持：

```cpp
#pragma once
```

优点是简单，不需要设计唯一宏名；缺点是它不是 C++ 标准的一部分，虽然主流编译器都支持。保护宏更标准、更可移植。

#### 常见误区

- 保护宏只能防止同一翻译单元内重复包含，不能解决多个 `.cpp` 中定义同一个全局变量导致的链接重复定义。
- 头文件中应尽量放声明、类型定义、模板定义、`inline` 函数，不要随意放普通函数或全局变量定义。

### 2. `#include "myhead1.h"` 和 `#include <myhead2.h>` 的处理和区别。

#### 面试回答

预处理阶段，编译器会处理 `#include` 指令，把指定头文件的内容展开到当前源文件中，再继续进行宏替换、条件编译等预处理工作。双引号形式通常先从当前源文件所在目录或工程指定目录查找，再查系统 include 目录；尖括号形式通常直接从系统头文件目录和编译器 `-I` 指定的目录查找。出现 `file not found` 时，应检查文件是否存在、路径大小写、include 路径配置、构建系统参数和头文件名是否写错。

#### 预处理做了什么

```cpp
#include "myhead1.h"
```

预处理器会把 `myhead1.h` 的内容插入到该行位置。可以用命令查看预处理结果：

```bash
g++ -E main.cpp -o main.i
```

#### 双引号和尖括号区别

| 写法 | 常见查找方式 | 典型用途 |
| --- | --- | --- |
| `#include "xxx.h"` | 先当前目录/工程目录，再系统目录 | 自己项目的头文件 |
| `#include <xxx.h>` | 系统目录、`-I` 指定目录 | 标准库、系统库、第三方库 |

实际查找顺序会受编译器、构建系统、`-I`、`-isystem` 等参数影响。

#### `file not found` 排查

1. 确认文件真实存在。
2. 确认大小写一致，Linux 文件系统区分大小写。
3. 确认 include 写法和目录层级正确。
4. 检查编译命令是否包含正确 `-I/path/to/include`。
5. 检查 CMake、Makefile、IDE include path 配置。
6. 检查是否把生成头文件漏生成或漏加入依赖。

#### 示例

```bash
g++ -I./include main.cpp -o app
```

此时可以：

```cpp
#include "foo.h"
#include <foo.h>
```

但工程头文件通常仍建议用双引号，表达语义更清楚。

### 3. C++ 为什么使用命名空间？什么时候用 `using`，什么时候用前缀？

#### 面试回答

C++ 使用命名空间是为了避免不同库、不同模块之间的名字冲突，并把相关代码组织到逻辑作用域中。比如标准库使用 `std` 命名空间，业务项目也可以使用 `company::module`。`using namespace` 会把整个命名空间的名字引入当前作用域，容易污染命名空间，尤其不应该出现在头文件中。公共接口、头文件和容易冲突的名字建议显式写前缀，如 `std::vector`；在 `.cpp` 文件或函数内部小范围可以使用 `using std::string` 这种精确 using。

#### 为什么需要命名空间

没有命名空间时，不同库可能定义同名类型或函数：

```cpp
class Logger {};
```

多个库都叫 `Logger` 就会冲突。用命名空间后：

```cpp
namespace net {
class Logger {};
}

namespace db {
class Logger {};
}
```

使用时：

```cpp
net::Logger net_logger;
db::Logger db_logger;
```

#### 不推荐头文件写 `using namespace std`

错误示例：

```cpp
// bad.h
#include <vector>
using namespace std;
```

任何包含 `bad.h` 的文件都会被迫引入整个 `std`，可能和业务名字冲突。

#### 推荐使用方式

头文件中：

```cpp
class UserRepo {
public:
    std::vector<int> ids() const;
};
```

源文件中可小范围 using：

```cpp
using std::string;
using std::vector;
```

函数内部也可以：

```cpp
void f() {
    using std::cout;
    cout << "hello\n";
}
```

#### 常见追问

- **命名空间能嵌套吗？**  
  可以。C++17 支持简写：

```cpp
namespace company::search {
class Index {};
}
```

### 4. `#define` 用法细节

#### 面试回答

`#define` 是预处理阶段的文本替换，不受 C++ 类型系统约束，没有作用域和类型检查；`typedef`/`using` 是类型别名，发生在编译阶段，受作用域和类型检查约束。宏函数容易出现优先级问题、多次求值副作用和调试困难，因此能用 `constexpr`、`inline` 函数、模板或 `using` 时应优先不用宏。

#### `#define` 和 `typedef/using` 区别

| 对比项 | `#define` | `typedef/using` |
| --- | --- | --- |
| 阶段 | 预处理 | 编译 |
| 本质 | 文本替换 | 类型别名 |
| 类型检查 | 无 | 有 |
| 作用域 | 宏定义后到取消或文件结束 | 遵循 C++ 作用域 |
| 调试 | 较困难 | 正常 |

#### 题目计算

```cpp
#define EQ(a) 0x12==a
EQ(12)
```

展开为：

```cpp
0x12 == 12
```

`0x12` 是十六进制，等于十进制 18，所以结果是 `false`，即 `0`。

```cpp
#define ADD(a) a*2
ADD(1+1)
```

展开为：

```cpp
1 + 1 * 2
```

乘法优先级高，所以结果是 `3`，不是 `4`。

```cpp
#define TH(a) (!a||a==-2)
TH(1)
```

展开：

```cpp
(!1 || 1 == -2)
```

结果是 `false`，即 `0`。

```cpp
TH(-2)
```

展开：

```cpp
(!-2 || -2 == -2)
```

`!-2` 为 `false`，`-2 == -2` 为 `true`，整体为 `true`，即 `1`。

#### 宏函数缺陷

1. 参数没有括号导致优先级错误。
2. 参数可能被多次求值。
3. 没有类型检查。
4. 调试时看不到真正函数调用。
5. 容易和作用域、命名冲突。

#### 改进

宏写法至少加括号：

```cpp
#define ADD(a) ((a) * 2)
```

但更推荐：

```cpp
constexpr int add2(int a) {
    return a * 2;
}
```

副作用示例：

```cpp
#define SQUARE(x) ((x) * (x))
int i = 3;
int y = SQUARE(++i); // ++i 被执行两次，危险
```

### 5. 字节计算（32 位编译器）。

#### 面试回答

这类题考察 `sizeof`、结构体对齐和指针大小。结构体大小不是成员大小简单相加，还要满足每个成员的对齐要求，并且整个结构体大小通常要是最大对齐成员大小的整数倍。32 位环境下普通指针大小通常是 4 字节，64 位环境下通常是 8 字节。

#### `struct A`

```cpp
struct A {
    unsigned int a; // 4
    int b;          // 4
};
```

布局：

```text
offset 0~3: a
offset 4~7: b
```

最大对齐为 4，成员总大小 8，已经是 4 的倍数：

```text
sizeof(A) = 8
```

#### `struct B`

```cpp
struct B {
    short a; // 2
    int b;   // 4
};
```

布局：

```text
offset 0~1: a
offset 2~3: padding
offset 4~7: b
```

`int` 需要按 4 字节对齐，所以 `short` 后补 2 字节：

```text
sizeof(B) = 8
```

#### 指针大小

```cpp
const char* p = "string";
sizeof(p)
```

`p` 是指针，不是数组。32 位编译器下：

```text
sizeof(p) = 4
```

64 位下通常为 8。

如果写成数组：

```cpp
const char arr[] = "string";
sizeof(arr) // 7，包含末尾 '\0'
```

#### 常见追问

- **如何减少结构体大小？**  
  可以按成员对齐从大到小排列，减少 padding。

```cpp
struct C {
    int b;
    short a;
};
```

### 6. 简述变量声明和定义的区别

#### 面试回答

声明告诉编译器某个名字的类型、作用域和链接信息，使后续代码可以使用这个名字；定义不仅声明名字，还真正创建实体，通常会分配存储空间。一个变量可以声明多次，但在同一链接范围内通常只能定义一次。函数原型是声明，带函数体的是定义；`extern int x;` 是变量声明，`int x;` 是变量定义。

#### 示例

```cpp
extern int g_count; // 声明，不分配存储
int g_count = 0;    // 定义，分配存储
```

函数：

```cpp
int add(int, int); // 声明

int add(int a, int b) { // 定义
    return a + b;
}
```

#### 头文件和源文件常见做法

头文件：

```cpp
// config.h
extern int g_port;
int add(int a, int b);
```

源文件：

```cpp
// config.cpp
int g_port = 8080;

int add(int a, int b) {
    return a + b;
}
```

#### 常见误区

- 在头文件直接写 `int g_port = 8080;`，被多个 `.cpp` 包含后会导致多重定义。
- C++17 起可以用 `inline` 变量放头文件，但要明确知道语义。

### 7. 在 C++ 文件中调用链接 C 语言代码函数，需要做什么特殊处理？

#### 面试回答

需要在 C++ 侧对 C 函数声明使用 `extern "C"`，告诉 C++ 编译器按 C 语言链接规则生成和查找符号名，避免 C++ 名字改编。C++ 支持函数重载，编译器会把函数名、参数类型等编码到符号名里；C 没有函数重载，符号名通常更直接。如果 C++ 调 C 函数不加 `extern "C"`，链接器可能找不到 C 目标文件中的符号。

#### C 头文件兼容写法

```c
// c_api.h
#ifndef C_API_H
#define C_API_H

#ifdef __cplusplus
extern "C" {
#endif

int c_add(int a, int b);

#ifdef __cplusplus
}
#endif

#endif
```

C 实现：

```c
// c_api.c
int c_add(int a, int b) {
    return a + b;
}
```

C++ 调用：

```cpp
#include "c_api.h"

int main() {
    return c_add(1, 2);
}
```

#### 常见追问

- **`extern "C"` 能修饰类吗？**  
  不能按 C ABI 导出 C++ 类语义。通常只用于普通函数和部分全局变量声明。

- **C 调 C++ 怎么办？**  
  在 C++ 中提供 `extern "C"` 包装函数，内部再调用 C++ 对象或函数。

### 8. 找出代码编译错误原因

#### 题目代码

```cpp
class A {
public:
    A():data(0){};
    ~A(){};
    int data;
    void inline inc(){data++;};
    static void dec(){data--;};
};
```

#### 面试回答

编译错误出在 `static void dec(){data--;};`。静态成员函数属于类本身，不属于某个具体对象，因此没有 `this` 指针，不能直接访问非静态成员变量 `data`。`data` 是每个对象各自拥有的一份成员，静态函数不知道应该修改哪个对象的 `data`。解决方式是把 `dec` 改成普通成员函数，或者给 `dec` 传入对象引用，或者把 `data` 也设计成静态成员。

#### 为什么 `inc` 可以

```cpp
void inline inc(){ data++; }
```

`inc` 是非静态成员函数，调用时必须依赖对象：

```cpp
A a;
a.inc();
```

编译器会隐式传入 `this` 指针，相当于：

```cpp
void inc(A* this) {
    this->data++;
}
```

#### 静态成员函数没有 `this`

```cpp
static void dec() {
    data--; // 错误，不知道 data 属于哪个对象
}
```

#### 修改方式

方式 1：改为普通成员函数。

```cpp
void dec() {
    data--;
}
```

方式 2：传入对象。

```cpp
static void dec(A& obj) {
    obj.data--;
}
```

方式 3：把数据改为静态成员。

```cpp
class A {
public:
    static int data;
    static void dec() { data--; }
};

int A::data = 0;
```

#### 补充

`void inline inc()` 的写法不常见，一般写成：

```cpp
inline void inc() { data++; }
```

类内定义的成员函数默认就是 inline 候选。

### 9. 简述值传递、指针传递的区别

#### 面试回答

值传递会把实参复制一份给形参，函数内部修改形参不会影响原对象，适合小对象或不需要修改实参的场景。指针传递传的是地址的副本，函数可以通过解引用修改原对象，也可以传 `nullptr` 表示没有对象。C++ 中还常用引用传递，语义上表示必须绑定到一个对象；对于大对象且不需要修改，推荐使用 `const T&` 避免拷贝。

#### 值传递

```cpp
void f(int x) {
    x = 10;
}

int a = 1;
f(a); // a 仍然是 1
```

#### 指针传递

```cpp
void g(int* p) {
    if (p) *p = 10;
}

int a = 1;
g(&a); // a 变成 10
```

#### 引用传递

```cpp
void h(int& x) {
    x = 10;
}

int a = 1;
h(a); // a 变成 10
```

#### 对比表

| 方式 | 是否复制对象 | 能否修改实参 | 能否为空 | 典型场景 |
| --- | --- | --- | --- | --- |
| 值传递 | 是 | 否 | 不涉及 | 小对象、只读 |
| 指针传递 | 复制地址 | 可以 | 可以为 `nullptr` | 可选对象、C 接口 |
| 引用传递 | 不复制对象 | 可以 | 通常不能为空 | 必须有对象 |
| `const T&` | 不复制对象 | 不可修改 | 通常不能为空 | 大对象只读 |

#### 常见追问

- **指针传递本身是不是值传递？**  
  是。指针变量本身也是按值复制，只是复制的是地址，函数通过地址能访问原对象。

### 10. `static` 加在变量、函数、成员函数上，本质共同点是什么？

#### 面试回答

`static` 在不同位置含义不同，但共同点是改变名字的生命周期、链接属性或归属关系，使其不再是普通自动局部变量、普通外部符号或普通对象成员。局部静态变量具有静态存储期，只初始化一次；文件作用域的 `static` 变量或函数具有内部链接，只在当前翻译单元可见；类的静态成员属于类本身，而不是某个对象；静态成员函数没有 `this` 指针，只能直接访问静态成员。

#### 四种常见用法

| 用法 | 含义 | 示例 |
| --- | --- | --- |
| 局部静态变量 | 静态存储期，函数内可见 | `static int count;` |
| 文件作用域静态变量 | 内部链接 | `static int g_value;` |
| 静态函数 | 内部链接，仅本 `.cpp` 可见 | `static void helper();` |
| 类静态成员 | 属于类，不属于对象 | `static int total;` |
| 静态成员函数 | 无 `this` 指针 | `static void f();` |

#### 局部静态变量

```cpp
int nextId() {
    static int id = 0;
    return ++id;
}
```

`id` 只初始化一次，函数返回后仍然存在。C++11 起局部静态变量初始化线程安全。

#### 文件作用域 static

```cpp
static void helper() {}
```

`helper` 只在当前 `.cpp` 内可见，避免和其他翻译单元同名函数冲突。现代 C++ 也常用匿名命名空间：

```cpp
namespace {
void helper() {}
}
```

#### 类静态成员

```cpp
class Counter {
public:
    static int total;
};

int Counter::total = 0;
```

所有对象共享同一份 `total`。

### 11. 计算循环结束后 `sum`

#### 题目代码

```cpp
unsigned int sum = 0;
for (unsigned int i = 0; i < 10; i++) {
    if (2 == ++i)
        continue;
    if (i == 5)
        break;
    if (i)
        sum++;
}
```

#### 面试回答

最终 `sum = 2`。关键是循环体内部先执行 `++i`，循环末尾还会执行 `i++`，所以 `i` 每轮不是简单加 1。第三轮中 `++i` 后 `i` 变成 5，触发 `break` 退出循环，因此只在前两轮执行了 `sum++`。

#### 推导过程

初始：

```text
sum = 0, i = 0
```

第一轮：

```text
i < 10 成立
++i -> i = 1
2 == 1 不成立
i == 5 不成立
i 非 0，sum++ -> sum = 1
循环末尾 i++ -> i = 2
```

第二轮：

```text
i = 2, i < 10 成立
++i -> i = 3
2 == 3 不成立
i == 5 不成立
i 非 0，sum++ -> sum = 2
循环末尾 i++ -> i = 4
```

第三轮：

```text
i = 4, i < 10 成立
++i -> i = 5
2 == 5 不成立
i == 5 成立，break
```

所以：

```text
sum = 2
```

#### 注意 `continue`

如果某轮执行 `continue`，仍然会执行 `for` 语句的第三部分 `i++`。本题没有触发 `continue`，因为 `++i` 没有等于 2。

### 12. 为什么指针能指向函数，并通过指针动态调用函数？

#### 面试回答

函数编译后会生成机器指令并放在代码段中，函数名在表达式中通常可以转换为函数入口地址。函数指针变量保存的就是这个入口地址，因此可以通过函数指针间接调用函数，本质是 CPU 跳转到该地址执行指令。C++ 中虚函数多态也可以理解为一种间接调用：对象通过虚指针找到虚表，虚表中保存函数指针，运行时根据实际对象类型调用对应函数。

#### 函数指针示例

```cpp
int add(int a, int b) {
    return a + b;
}

int sub(int a, int b) {
    return a - b;
}

int compute(int a, int b, int (*op)(int, int)) {
    return op(a, b);
}

int x = compute(3, 2, add); // 5
int y = compute(3, 2, sub); // 1
```

#### 为什么能动态调用

```text
函数指针 op 保存 add 或 sub 的入口地址
compute 执行 op(a,b)
运行时跳转到 op 指向的函数
```

#### 和多态的关系

虚函数调用大致流程：

```text
对象地址 -> vptr -> vtable -> function pointer -> call
```

函数指针是显式保存函数地址；虚函数是编译器生成和维护函数指针表。

#### 常见追问

- **普通函数指针和成员函数指针一样吗？**  
  不一样。非静态成员函数有隐含 `this`，成员函数指针表示更复杂，调用时需要对象。

### 13. 指针计算（32 位编译器）

#### 题目代码

```cpp
void func() {
    unsigned char tmp[10] = {0};
    tmp[0] = 0;
    tmp[1] = 1;
    tmp[2] = 2;
    tmp[3] = 3;

    unsigned char *p = tmp;
    p++;
    //*p=?

    unsigned short *sp = (unsigned short *)tmp;
    sp++;
    //*sp=?
}
```

#### 面试回答

`p` 是 `unsigned char*`，`p++` 会前进 1 个字节，因此指向 `tmp[1]`，所以 `*p = 1`。`sp` 是 `unsigned short*`，`sp++` 会前进 `sizeof(unsigned short)`，通常是 2 个字节，因此指向从 `tmp[2]` 开始的两个字节。若按常见 x86 小端环境解释，低地址 `tmp[2]=2` 是低字节，`tmp[3]=3` 是高字节，所以 `*sp = 0x0302 = 770`。

#### 内存布局

```text
地址偏移:  0    1    2    3
tmp 值:   0x00 0x01 0x02 0x03
```

`unsigned char*`：

```text
p = &tmp[0]
p++ -> &tmp[1]
*p = 0x01 = 1
```

`unsigned short*`：

```text
sp = reinterpret_cast<unsigned short*>(&tmp[0])
sp++ -> 地址 + 2 -> &tmp[2]
```

小端读取两个字节：

```text
低字节 tmp[2] = 0x02
高字节 tmp[3] = 0x03
值 = 0x0302 = 770
```

#### 重要注意

> [!CAUTION]
> 这道题默认 32 位 x86 小端且允许非严格对齐访问。严格来说，把 `unsigned char*` 转成 `unsigned short*` 再解引用可能涉及对齐要求和严格别名规则问题，在某些平台上可能未定义或结果不同。面试按题目语境通常回答小端结果 770，并补充端序和对齐风险。

### 14. C++ 引入模板相比 C 的优势

#### 面试回答

C++ 模板提供了类型安全的泛型编程能力，可以写一份代码适配多种类型，并在编译期生成对应类型的实现。相比 C 中常用的 `void*` 和宏，模板有类型检查，能避免大量强制转换和运行时错误；模板还支持零开销抽象，编译器可以内联和优化。STL 的容器、迭代器和算法就是模板泛型编程的典型应用。

#### C 的泛型方式问题

宏：

```c
#define MAX(a, b) ((a) > (b) ? (a) : (b))
```

问题：

- 没有类型检查。
- 参数可能多次求值。
- 调试困难。

`void*`：

```c
void sort(void* base, size_t n, size_t size, int (*cmp)(const void*, const void*));
```

问题：

- 类型信息丢失。
- 使用者要自己转换。
- 错误可能到运行时才暴露。

#### C++ 模板

```cpp
template <class T>
T maxValue(const T& a, const T& b) {
    return a < b ? b : a;
}
```

优点：

- 类型安全。
- 编译期实例化。
- 可内联优化。
- 支持自定义类型。
- 支持泛型容器和算法。

#### 常见追问

- **模板有什么缺点？**  
  编译时间增加，错误信息复杂，模板定义通常要放头文件，过度实例化可能导致代码膨胀。

### 15. 计算 `switch` 后 `m`

#### 题目代码

```cpp
u32 k = 7, m = 0;
switch (k) {
case 1:
    m++;
    break;
case 7:
    m = 1;
default:
    m = 0;
}
```

#### 面试回答

最终 `m = 0`。因为 `k = 7`，程序进入 `case 7`，先执行 `m = 1`，但是 `case 7` 后面没有 `break`，所以会继续向下执行 `default` 分支，把 `m` 重新赋值为 0。这就是 `switch` 的 fall-through 机制。

#### 执行过程

```text
k = 7
进入 case 7
m = 1
没有 break
继续执行 default
m = 0
switch 结束
```

#### 如何避免

```cpp
switch (k) {
case 1:
    m++;
    break;
case 7:
    m = 1;
    break;
default:
    m = 0;
    break;
}
```

#### C++17 标注意图

如果确实需要贯穿，可以写：

```cpp
case 7:
    m = 1;
    [[fallthrough]];
default:
    m = 0;
```

这样能让编译器和读代码的人知道这是有意为之。

### 16. 举例三种常见内存泄漏/溢出情景

#### 面试回答

常见内存问题包括：动态分配后没有释放导致内存泄漏；数组越界写导致缓冲区溢出；释放后继续使用导致 use-after-free；重复释放导致 double free；递归过深或栈上大对象导致栈溢出。工程中应优先使用 RAII、智能指针、标准容器，并配合 ASan、Valgrind、静态分析和代码审查发现问题。

#### 1. 内存泄漏

```cpp
void leak() {
    int* p = new int[100];
    if (someError()) return; // delete[] p 未执行
    delete[] p;
}
```

改进：

```cpp
void noLeak() {
    std::vector<int> v(100);
    if (someError()) return;
}
```

或：

```cpp
auto p = std::make_unique<int[]>(100);
```

#### 2. 缓冲区溢出

```cpp
char buf[8];
strcpy(buf, "this string is too long");
```

越界写会破坏栈上的其他数据，可能导致崩溃或安全漏洞。

改进：

```cpp
std::string s = "this string is too long";
```

#### 3. Use-after-free

```cpp
int* p = new int(10);
delete p;
*p = 20; // 已释放后继续使用
```

释放后指针应不再使用，裸指针所有权要尽量用智能指针表达。

#### 4. 栈溢出

```cpp
void f() {
    int big[10000000]; // 栈空间可能不够
}
```

递归过深也会栈溢出：

```cpp
void dfs() {
    dfs();
}
```

#### 检测工具

```bash
g++ -fsanitize=address -g main.cpp -o app
valgrind --leak-check=full ./app
```

### 17. 变量计算（32 位编译器）

#### 题目代码

```cpp
unsigned short a = 0, b;

void func(void)
{
    /* 此时 b=? */
    unsigned short a = 2;
    a += 2;
    /* a=? */

    b = 21;
    b /= 2;
    /* b=? */

    unsigned short c = 0x0001;
    c = c << 8;
    /* c=? */

    unsigned short d;
    /* d 的内存分配在哪里? 此时 d 的值为? */
    short tmp = -100;
    d = tmp;
    /* 此时 d 的值为? */

    unsigned char e = 10;
    e = e > 0 ? 1 : 0;
    /* 此时 e 的值为? */

    u8 f = 0x01;
    f &= 0x10;
    /* 此时 f 的值为? */
}
```

#### 面试回答

全局变量 `b` 具有静态存储期，会被零初始化，所以函数开始时 `b = 0`。函数内部的 `unsigned short a = 2` 屏蔽了全局 `a`，执行 `a += 2` 后局部 `a = 4`。`b = 21; b /= 2;` 是整数除法，所以 `b = 10`。`c = 0x0001 << 8` 后为 `0x0100`，即 256。局部变量 `d` 分配在栈上，未初始化时值不确定；`d = tmp` 时 `tmp = -100` 转为 `unsigned short`，按模 `2^16` 得到 65436。`e > 0` 为真，所以 `e = 1`。`f = 0x01; f &= 0x10` 后为 0。

#### 逐项结果

| 位置 | 结果 | 原因 |
| --- | --- | --- |
| 函数开始 `b` | `0` | 全局变量零初始化 |
| 局部 `a += 2` 后 | `4` | 局部 `a` 初始 2，加 2 |
| `b /= 2` 后 | `10` | `21 / 2` 整数除法 |
| `c << 8` 后 | `256` | `0x0001 << 8 = 0x0100` |
| 未初始化 `d` | 不确定 | 栈上自动变量未初始化 |
| `d = tmp` 后 | `65436` | `-100` 转 `uint16_t` |
| `e` | `1` | 条件表达式为真 |
| `f` | `0` | `0x01 & 0x10 = 0x00` |

#### 关于 `d = tmp`

`unsigned short` 通常是 16 位，取值范围 0 到 65535。`-100` 转换为 `unsigned short`：

```text
2^16 - 100 = 65536 - 100 = 65436
```

十六进制为：

```text
0xFF9C
```

#### 变量遮蔽

全局：

```cpp
unsigned short a = 0;
```

局部：

```cpp
unsigned short a = 2;
```

局部 `a` 会遮蔽全局 `a`，函数中直接写 `a` 指局部变量。若要访问全局 `a`：

```cpp
::a
```

### 18. 简述 C/C++ 编译过程，涉及 `.h`、`.c`、`.cpp`、`.o`、`.a`

#### 面试回答

C/C++ 从源代码到可执行文件通常经过预处理、编译、汇编和链接四个阶段。`.h` 是头文件，提供声明、宏、模板等内容；`.c` 是 C 源文件，`.cpp` 是 C++ 源文件；每个源文件经过预处理、编译和汇编后生成目标文件 `.o`；静态库 `.a` 是多个 `.o` 文件的归档；链接阶段把多个 `.o`、`.a` 以及动态库符号解析到一起，生成可执行文件。

#### 四个阶段

1. 预处理：
   - 展开 `#include`。
   - 展开宏。
   - 处理条件编译。
   - 去掉注释。

```bash
g++ -E main.cpp -o main.i
```

2. 编译：
   - 把预处理后的 C++ 转成汇编。
   - 做语法检查、语义分析、优化。

```bash
g++ -S main.i -o main.s
```

3. 汇编：
   - 把汇编转成目标文件 `.o`。

```bash
g++ -c main.s -o main.o
```

4. 链接：
   - 合并目标文件。
   - 解析外部符号。
   - 链接静态库或动态库。
   - 生成可执行文件。

```bash
g++ main.o util.o -L. -lfoo -o app
```

#### 文件类型说明

| 文件 | 作用 |
| --- | --- |
| `.h/.hpp` | 头文件，声明、模板、inline 实现 |
| `.c` | C 源文件 |
| `.cpp/.cc` | C++ 源文件 |
| `.o/.obj` | 目标文件，包含机器码和符号表 |
| `.a/.lib` | 静态库，本质是 `.o` 归档 |
| `.so/.dll` | 动态库，运行或加载时链接 |

#### 静态库 `.a`

创建静态库：

```bash
g++ -c add.cpp -o add.o
g++ -c sub.cpp -o sub.o
ar rcs libcalc.a add.o sub.o
```

使用：

```bash
g++ main.o -L. -lcalc -o app
```

链接器会从 `.a` 中提取需要的 `.o` 合入可执行文件。

#### 常见追问

- **头文件会被单独编译吗？**  
  通常不会。头文件被源文件 `#include` 后作为源文件的一部分参与编译。

- **undefined reference 是哪个阶段错误？**  
  链接阶段错误，通常是声明了函数但没有找到定义，或库链接顺序/路径不对。

### 19. 英语测试：翻译任选三个功能特性

#### 面试回答

这题主要考察技术英语理解。原文来自某个 JSON 库的设计目标，可以任选三个功能特性翻译。下面给出多个特性的准确翻译，面试时任选三个即可。

#### Intuitive syntax

原意：

```text
Intuitive syntax. In languages such as Python, JSON feels like a first class data type.
We used all the operator magic of modern C++ to achieve the same feeling in your code.
```

翻译：

**直观的语法。** 在 Python 这类语言中，JSON 用起来就像一等数据类型一样自然。这个库使用现代 C++ 的运算符重载等能力，让你在 C++ 代码中也能获得类似的自然体验。看一下示例代码，就能理解作者想表达的效果。

#### Trivial integration

原意：

```text
Trivial integration. Our whole code consists of a single header file json.hpp.
That's it. No library, no subproject, no dependencies, no complex build system.
```

翻译：

**极其简单的集成方式。** 整个库只有一个头文件 `json.hpp`，仅此而已。不需要额外链接库，不需要加入子项目，没有外部依赖，也不需要复杂的构建系统。该类使用标准 C++11 编写，通常不需要调整编译器参数或项目配置。

#### Serious testing

原意：

```text
Serious testing. Our class is heavily unit-tested and covers 100% of the code,
including all exceptional behavior.
```

翻译：

**严格的测试。** 这个库经过了大量单元测试，测试覆盖率达到 100%，包括各种异常行为。此外，项目还使用 Valgrind 和 Clang Sanitizers 检查，确保没有内存泄漏。Google OSS-Fuzz 也会持续对所有解析器进行模糊测试，相当于长期执行海量测试，以保持项目质量。

#### Memory efficiency

翻译：

**内存效率。** 每个 JSON 对象会有一个指针大小的额外开销，也就是联合体的最大大小，以及一个枚举元素的开销。默认的通用实现使用 `std::string` 表示字符串，用 `int64_t`、`uint64_t` 或 `double` 表示数字，用 `std::map` 表示对象，用 `std::vector` 表示数组，用 `bool` 表示布尔值。不过，用户也可以通过模板把通用的 `basic_json` 类定制成自己需要的形式。

#### Speed

翻译：

**速度。** 当然，市面上肯定存在更快的 JSON 库。不过，如果你的目标是通过加入一个单头文件来快速获得 JSON 支持、提升开发效率，那么这个库就是合适的选择。只要你知道如何使用 `std::vector` 或 `std::map`，就已经具备使用它的基础。

#### 面试技巧

- 翻译技术文档时不要逐字硬翻，要保留技术含义。
- `first class data type` 可译为“一等数据类型”。
- `vanilla C++11` 可译为“标准/原生 C++11，不依赖特殊扩展”。
- `fuzz tests` 是“模糊测试”，不是“模糊的测试”。
