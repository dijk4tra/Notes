# C/C++基础

## 基础语法、内存与编译

### 1. 描述 C++ 程序的内存由哪几个部分组成，每个区域分别有什么作用和特点？

#### 经典回答

从程序员理解和使用内存的角度来看，C++ 程序运行时的内存通常可以分为：代码区、全局/静态区、常量区、堆区和栈区。不同区域存放的数据类型、生命周期和管理方式都不同。

- **代码区**用于存放程序编译后的机器指令，也就是函数代码。代码区通常是只读、可执行的，防止程序运行过程中代码被意外修改。
- **全局/静态区**用于存放全局变量和静态变量，包括已初始化和未初始化的全局变量、静态变量。这些变量在程序开始运行时分配，在程序结束时释放，生命周期贯穿整个程序运行期间。
- **常量区**用于存放常量数据，例如字符串字面量、被编译器放入只读区域的常量等。常量区通常只读，强行修改字符串字面量等内容可能导致程序崩溃。
- **堆区**用于动态内存分配，例如通过 `new`、`malloc` 申请的内存。堆区内存由程序员手动释放，或者通过智能指针、RAII 机制自动管理。如果管理不当，容易出现内存泄漏、野指针、重复释放等问题。
- **栈区**用于函数调用，存放局部变量、函数参数、返回地址、调用上下文等。栈区由系统自动分配和释放，效率较高，但空间有限，递归过深或局部变量过大可能导致栈溢出。

| 区域        | 典型内容                              | 生命周期         | 权限/特点                                                |
| ----------- | ------------------------------------- | ---------------- | -------------------------------------------------------- |
| 栈区        | 局部变量、函数参数、返回地址、调用帧  | 函数调用期间     | 自动分配和释放，速度快，但空间有限，递归过深会栈溢出     |
| 堆区        | `new` / `malloc` 动态申请的对象或内存 | 从申请到释放     | 由程序员或 RAII 管理，容易出现内存泄漏、野指针、重复释放 |
| 全局/静态区 | 全局变量、静态变量                    | 程序整个生命周期 | 程序启动时分配，程序结束时释放                           |
| 常量区      | 字符串字面量、常量数据                | 程序整个生命周期 | 通常只读，强行修改可能导致段错误                         |
| 代码区      | 函数代码、编译后的机器指令            | 程序整个生命周期 | 通常只读、可执行                                         |

```text
高地址
↓
栈区
堆区
全局/静态区   ≈ .data + .bss
常量区        ≈ .rodata
代码区        ≈ .text
↓
低地址
```

```cpp
// 代码示例

int g1 = 1;              // 全局/静态区：已初始化的全局变量
int g2;                  // 全局/静态区：未初始化的全局变量
static int g3 = 3;       // 全局/静态区：静态变量

const char* p = "abc";   // p 本身在全局/静态区，"abc" 在常量区

void func() {            // 函数代码在代码区
    static int s = 10;   // 全局/静态区：局部静态变量
    int a = 20;          // 栈区：局部变量

    int* q = new int(30); // q 在栈区，*q 指向的对象在堆区
    delete q;            // 释放堆区内存
}

int main() {             // main 函数代码在代码区
    int x = 10;          // 栈区
    func();
    return 0;
}
```

#### 系统回答

从操作系统和进程虚拟地址空间的角度看，C++ 程序运行后看到的是一个进程的虚拟地址空间，通常可以分为：`.text`、`.rodata`、`.data`、`.bss`、heap、stack、mmap 等区域。

- `.text` 段是代码段，存放编译后的机器指令，通常具有只读、可执行权限，可以被多个进程共享。

- `.rodata` 段是只读数据段，通常存放字符串字面量、只读常量表、虚表等只读数据。它一般只读，不允许写入。

- `.data` 段存放已经显式初始化的全局变量和静态变量，例如：

  ```cpp
  int g = 10;
  static int s = 20;
  ```

- `.bss` 段存放未初始化或者零初始化的全局变量和静态变量，例如：

  ```cpp
  int g;
  static int s;
  ```

- `.bss` 在可执行文件中通常只记录大小，程序加载时由系统清零。

- `heap` 是堆区，用于动态内存分配，例如 `malloc`、`new` 申请的内存。它的生命周期从申请开始，到 `free` 或 `delete` 释放结束，通常由运行时内存分配器管理。

- `stack` 是栈区，用于函数调用，保存局部变量、函数参数、返回地址、寄存器现场等。栈由系统自动管理，函数调用时增长，函数返回时回收。

- `mmap` 区域用于内存映射，例如动态库加载、文件映射、匿名映射、大块内存分配等。很多动态库、共享内存和较大的 `malloc` 分配都可能通过 `mmap` 实现。

| 区域                 | 典型内容                               | 生命周期         | 权限/特点                                  |
| -------------------- | -------------------------------------- | ---------------- | ------------------------------------------ |
| 栈 stack             | 局部变量、函数参数、返回地址、调用帧   | 函数调用期间     | 自动分配释放，空间有限，递归过深会栈溢出   |
| mmap 区              | 动态库、文件映射、匿名映射、大块分配   | 映射期间         | 可用于共享内存、大文件访问、动态链接库加载 |
| 堆 heap              | `malloc/new` 申请的对象                | 从申请到释放     | 程序员或 RAII 控制，易泄漏、碎片、重复释放 |
| BSS `.bss`           | 未初始化或零初始化的全局/静态变量      | 进程整个生命周期 | 文件中只记录大小，加载时清零               |
| 已初始化数据 `.data` | 显式初始化的全局变量、静态变量         | 进程整个生命周期 | 可读写，程序加载时从可执行文件装入         |
| 只读数据 `.rodata`   | 字符串字面量、只读常量表、虚表常见位置 | 进程整个生命周期 | 只读，写入通常触发段错误                   |
| 代码段 `.text`       | 编译后的机器指令                       | 进程整个生命周期 | 通常只读、可执行，可被多个进程共享         |

```text
高地址
↓
stack     栈区
mmap      内存映射区
heap      堆区
.bss      BSS段
.data     数据段载
.rodata   只读数据段
.text     代码段
↓
低地址
```

```cpp
// 下面以典型 Linux/ELF 进程虚拟地址空间为例说明，实际内存布局会受到编译器、链接器、操作系统和优化级别影响
#include <cstdlib>

#ifdef __linux__
#include <sys/mman.h>
#include <unistd.h>
#endif

// -------------------------
// 全局/静态存储期对象
// -------------------------

int g1 = 1;              // .data：已初始化的全局变量
int g2;                  // .bss：未初始化的全局变量，程序加载时清零
int g3 = 0;              // .bss：零初始化的全局变量，通常也放在 .bss

static int sg1 = 3;      // .data：已初始化的静态变量，内部链接
static int sg2;          // .bss：未初始化的静态变量

const int cg = 100;      // 通常在 .rodata，若被优化掉则可能不实际占存储空间

const char* gp = "abc";
// gp 本身是全局指针变量，通常在 .data
// "abc" 是字符串字面量，通常在 .rodata

const char ro_arr[] = "hello";
// ro_arr 是全局 const 数组，通常整体放在 .rodata


// -------------------------
// 函数代码
// -------------------------

void func() {
    // func 的机器指令在 .text 代码段

    static int s1 = 10;  // .data：已初始化的局部静态变量
    static int s2;       // .bss：未初始化的局部静态变量

    int x = 10;          // stack：局部变量
    int arr[10];         // stack：局部数组

    int* p = new int(20);
    // p 本身是局部变量，在 stack 上
    // *p 指向的 int 对象在 heap 上

    delete p;
}


// -------------------------
// main 函数
// -------------------------

int main() {
    int a = 1;           // stack：局部变量
    const char* p = "xyz";
    // p 是局部指针变量，在 stack 上
    // "xyz" 是字符串字面量，通常在 .rodata

    int* q = new int(30);
    // q 在 stack 上
    // *q 在 heap 上

    delete q;

#ifdef __linux__
    void* mem = mmap(
        nullptr,
        4096,
        PROT_READ | PROT_WRITE,
        MAP_PRIVATE | MAP_ANONYMOUS,
        -1,
        0
    );

    // mem 指向 mmap 区域，属于进程虚拟地址空间中的内存映射区
    // mmap 区域常用于动态库、文件映射、匿名映射、大块内存分配等

    if (mem != MAP_FAILED) {
        munmap(mem, 4096);
    }
#endif

    return 0;
}
```

> [!CAUTION]
> 从程序员的视角来看，C++ 内存通常分为代码区、全局/静态区、常量区、堆区和栈区；
>
> 从操作系统和可执行文件装载的角度看，进程虚拟地址空间会进一步对应到.text、.rodata、.data、.bss、heap、stack、mmap 等区域。
>
> 前者是程序员视角的抽象分类，后者是更接近系统实现的分类。

### 2. 什么时候分配内存会产生内存碎片？

#### 面试回答

内存碎片通常出现在**堆区的动态内存分配**中，尤其是程序频繁 `malloc/free` 或 `new/delete`，并且申请的内存块大小不同、生命周期不同、释放顺序和申请顺序交错时。

内存碎片一般分为两类：**内部碎片**和**外部碎片**。

**内部碎片**是指分配器实际分配的内存比程序申请的内存更大，多出来的部分无法被有效使用。例如程序申请 13 字节，但由于内存对齐或分配器按固定规格分配，实际可能分配 16 字节或 32 字节，多出来的部分就是内部碎片。

**外部碎片**是指空闲内存总量足够，但这些空闲空间分散在不同位置，无法组成一块足够大的连续内存。比如堆上有很多小的空闲块，总共剩余 1MB，但没有一块连续的 512KB 空间，此时申请 512KB 仍然可能失败。

所以，内存碎片不是简单地“分配内存就会产生”，而是通常在**长期、频繁、大小不一、生命周期交错的动态分配和释放过程中逐渐产生**。

| 碎片类型 | 产生时机       | 原因                             | 举例                           |
| -------- | -------------- | -------------------------------- | ------------------------------ |
| 内部碎片 | 分配内存时     | 对齐、size class、固定块大小分配 | 申请 13 字节，实际分配 16 字节 |
| 外部碎片 | 多次分配释放后 | 空闲块不连续，无法合并成大块     | 总空闲 1MB，但没有连续的 512KB |

#### 典型场景

常见容易产生内存碎片的情况有：

1. **频繁申请和释放小对象**

​	例如服务端程序中频繁创建请求对象、消息对象、节点对象，如果每个对象都单独 `new/delete`，堆上就会不断产生小块分配和释放，容易形成碎片。

2. **对象大小差异很大**

​	如果程序一会儿申请几十字节的小对象，一会儿申请几 KB 或几 MB 的大对象，小块释放后不一定能被大块复用，容易产生外部碎片。

3. **长生命周期对象和短生命周期对象混合分配**

​	这是很常见的碎片来源。比如长生命周期对象一直占着中间位置，短生命周期对象被频繁创建和释放，就可能在长生命周期对象之间留下很多空洞。

4. **分配器为了对齐或性能按规格分配**

​	很多内存分配器不是你申请多少就刚好给多少，而是按 8 字节、16 字节、32 字节、64 字节等规格分配。这样可以提高分配效率，但会带来一定的内部碎片。

5. **容器频繁扩容或大量节点式分配**

​	例如频繁向 `std::vector` 追加元素但没有提前 `reserve`，可能导致多次重新分配；而 `std::list`、`std::map` 这类节点式容器，每个节点通常单独分配，也可能增加堆碎片压力。

#### 代码示例

```cpp
int* a = new int[100];      // 分配一块内存
int* b = new int[200];      // 分配一块内存
int* c = new int[100];      // 分配一块内存

delete[] b;                 // 中间释放，留下一个空洞

int* d = new int[300];      // 如果 b 留下的空间不够大，
                            // 分配器可能无法复用这个空洞，
                            // 只能去堆的其他位置找连续空间
```

这里 `b` 被释放后，堆中间留下了一块空闲区域。如果后续申请的 `d` 比这个空闲区域大，那么这块空闲空间就暂时无法被利用，随着类似情况增多，就会形成外部碎片。

#### 如何减少内存碎片

可以从几个方向优化：

- 使用对象池、内存池，适合大量同类型小对象。
- 按对象大小或生命周期分区分配，避免长短生命周期对象混在一起。
- 减少频繁的 `new/delete`，尽量复用对象。
- 对 `std::vector` 提前使用 `reserve`，减少扩容次数。
- 使用 RAII 和智能指针，避免泄漏和重复释放。
- 服务端高并发场景可以考虑合适的分配器，比如 jemalloc、tcmalloc。
- 对批量对象使用批量申请、批量释放，降低堆分配压力。

> [!NOTE]
> 内存碎片主要发生在堆区的动态内存分配中，尤其是频繁申请和释放不同大小、不同生命周期的内存块时。碎片分为内部碎片和外部碎片。内部碎片是因为内存对齐或分配器按固定规格分配，导致实际分配空间大于请求空间；外部碎片是因为空闲内存被分散成很多不连续的小块，虽然总量足够，但无法满足较大的连续内存申请。减少碎片的方法包括使用对象池、内存池、按对象类型或生命周期分区分配、减少频繁动态分配、提前 `reserve` 容器空间，以及在服务端场景选择更适合的内存分配器。

### 3. 负数的编码方式是什么？简述一下它的原理。

#### 面试回答

现代计算机的有符号整数通常使用补码表示。正数的补码和原码相同，负数的补码是其绝对值二进制按位取反再加 1。补码的好处是加法和减法可以统一成加法电路，并且 0 只有一种表示。

#### 详细解释

以 8 位整数为例：

```text
 5 原码/补码: 0000 0101
-5 的生成:
  0000 0101
  取反: 1111 1010
  加 1: 1111 1011
```

8 位有符号补码范围是 `[-128, 127]`。为什么负数多一个？因为 `1000 0000` 表示 `-128`，没有对应的 `+128`。补码判断负数也很简单：最高位为 1 通常表示负数。

> [!NOTE]
> C++20 起标准进一步贴近二进制补码现实，但涉及有符号溢出时仍要谨慎。`int` 溢出在 C++ 中是未定义行为。

### 4. 浮点数的编码方式是什么？简述一下它的原理。

#### 面试回答

主流浮点数遵循 IEEE 754，用符号位、指数位和尾数位表示。`float` 通常是 32 位：1 位符号、8 位指数、23 位尾数；`double` 通常是 64 位：1 位符号、11 位指数、52 位尾数。规格化数可理解为 `(-1)^sign * 1.fraction * 2^(exponent-bias)`。

#### 详细解释

浮点数并不是直接保存十进制小数，而是保存科学计数法的二进制形式：

| 类型 | 符号位 | 指数位 | 尾数位 | 指数偏移 |
| --- | ---: | ---: | ---: | ---: |
| `float` | 1 | 8 | 23 | 127 |
| `double` | 1 | 11 | 52 | 1023 |

它能表示非常大或非常小的数，但很多十进制小数不能被精确表示，例如 `0.1`。浮点还包含特殊值：`+0/-0`、无穷大、NaN、非规格化数。

```cpp
double a = 0.1 + 0.2;
double b = 0.3;
// 不建议 a == b
if (std::fabs(a - b) < 1e-12) {
    // 认为相等
}
```

### 5. 可执行程序是如何生成的？

#### 面试回答

C/C++ 源文件生成可执行文件通常经历预处理、编译、汇编、链接四个阶段。预处理展开头文件、宏和条件编译；编译把源代码转为汇编；汇编生成目标文件 `.o`；链接解析符号、合并段、处理重定位并链接静态库或动态库，最终生成 ELF/PE/Mach-O 等可执行文件。

```mermaid
flowchart LR
  A[.c/.cpp 源文件] --> B[预处理 .i/.ii]
  B --> C[编译 .s 汇编]
  C --> D[汇编 .o 目标文件]
  D --> E[链接 可执行文件]
```

#### 常见追问

- `.h` 本身不单独编译，而是在预处理阶段被包含进源文件。
- `.o` 中有代码、数据、符号表、重定位信息。
- 静态链接会把需要的库代码合入可执行文件；动态链接只记录依赖，运行时由动态链接器加载。
- 链接错误常见原因是只有声明没有定义、函数签名不一致、库顺序不对、C/C++ ABI 名字改编不一致。

### 6. 可执行程序是如何变成进程的？

#### 面试回答

程序是磁盘上的静态文件，进程是程序运行起来后的资源和执行状态。Linux 下通常由 shell `fork()` 出子进程，再由子进程 `execve()` 加载可执行文件。内核读取 ELF 头，建立虚拟地址空间，映射代码段、数据段、动态链接器和栈，设置入口地址、参数、环境变量和初始寄存器，随后交给调度器运行。

#### 详细流程

1. shell 解析命令，调用 `fork()` 创建子进程。
2. 子进程调用 `execve(path, argv, envp)`。
3. 内核检查权限和文件格式，读取 ELF Program Header。
4. 建立新的进程地址空间，映射可执行文件各段。
5. 若是动态链接程序，加载动态链接器并由其加载 `.so`。
6. 初始化用户栈，放入参数、环境变量和辅助向量。
7. 设置 PC/IP 到入口地址，进程进入可运行状态。

### 7. 在 C 语言中如何调用 C++ 函数？

#### 面试回答

C 不能直接调用普通 C++ 函数，因为 C++ 支持函数重载、命名空间、类等特性，编译后符号名会发生 name mangling。解决方式是在 C++ 侧提供 `extern "C"` 包装接口，导出 C ABI，C 代码只调用这些包装函数。

```cpp
// api.h
#ifdef __cplusplus
extern "C" {
#endif
int add_c_api(int a, int b);
#ifdef __cplusplus
}
#endif
```

```cpp
// api.cpp
int add_impl(int a, int b) {
    return a + b;
}

extern "C" int add_c_api(int a, int b) {
    return add_impl(a, b);
}
```

```c
// main.c
#include "api.h"
int x = add_c_api(1, 2);
```

> [!CAUTION]
> `extern "C"` 只能修饰函数或变量的链接方式，不能让 C 直接理解 C++ 类、模板、异常、重载。跨语言接口要尽量使用 C 可表达的类型。

### 8. 请描述几种常见的 C/C++ 缺陷和陷阱？

#### 面试回答

C/C++ 的主要陷阱来自手动资源管理、未定义行为和复杂类型系统。常见问题有野指针、悬空指针、内存泄漏、重复释放、数组越界、未初始化变量、整数溢出、宏副作用、对象切片、异常安全不足、虚析构遗漏、并发数据竞争等。现代 C++ 的防范思路是 RAII、智能指针、标准容器、类型安全转换和工具检测。

#### 分类说明

| 类型 | 典型问题 | 解决思路 |
| --- | --- | --- |
| 内存 | 泄漏、越界、double free、use-after-free | RAII、智能指针、ASan/Valgrind |
| 指针 | 空指针、野指针、悬空指针 | 初始化为 `nullptr`，明确所有权 |
| 字符串 | `strcpy/sprintf` 缓冲区溢出 | `std::string`、`snprintf` |
| 宏 | 优先级、副作用、多次求值 | `constexpr`、`inline`、模板 |
| 类型 | C 风格强转破坏类型安全 | `static_cast/dynamic_cast` 等 |
| OOP | 对象切片、非虚析构 | 使用引用/指针，多态基类虚析构 |

### 9. 重写、重载、重定义这三者有什么区别？

#### 面试回答

重载、重写、重定义都是“函数同名”相关概念，但发生位置、匹配规则和绑定时机完全不同。重载发生在同一作用域，函数名相同但参数列表不同，属于编译期多态；重写发生在继承体系中，派生类重新实现基类的虚函数，属于运行期多态；重定义也叫隐藏，发生在派生类和基类之间，只要派生类声明了与基类同名的成员，就会隐藏基类同名成员，不要求基类函数是虚函数，也不要求参数列表相同。

#### 1. 重载 overload

重载是指在同一作用域内，多个函数具有相同函数名，但参数列表不同。参数列表不同包括参数个数不同、参数类型不同、参数顺序不同。函数返回值不同不能单独构成重载，因为调用时编译器通常无法仅根据返回值判断应该调用哪个函数。

重载的特点：

- 发生在同一作用域。
- 函数名相同。
- 参数列表必须不同。
- 返回值类型可以不同，但不能只靠返回值区分。
- 编译期根据实参类型和重载决议规则确定调用哪个函数。
- 本质是静态多态，也就是编译期多态。

```cpp
void print(int x);
void print(double x);
void print(const std::string& s);

// int foo();
// double foo(); // 错误：仅返回值不同，不能构成重载
```

重载常用于提供统一语义的多种输入形式，例如构造函数重载、`print` 函数支持多种类型、`operator+` 支持不同参数组合等。

#### 2. 重写 override

重写是指派生类重新实现基类中的虚函数。它必须发生在继承关系中，并且基类函数必须是虚函数。派生类函数的函数名、参数列表、cv 限定、引用限定等需要与基类虚函数匹配；返回值通常也要相同，但协变返回类型是例外。

重写的特点：

- 发生在基类和派生类之间。
- 基类函数必须是 `virtual`。
- 派生类函数签名必须匹配基类虚函数。
- 通过基类指针或基类引用调用时，根据对象实际类型动态分派。
- 本质是动态多态，也就是运行期多态。
- C++11 起建议显式写 `override`，让编译器检查是否真的重写成功。

```cpp
struct Base {
    virtual void run(int x) {
        std::cout << "Base::run\n";
    }
    virtual ~Base() = default;
};

struct Derived : Base {
    void run(int x) override {
        std::cout << "Derived::run\n";
    }
};

Base* p = new Derived;
p->run(1); // 调用 Derived::run，运行期绑定
delete p;
```

如果派生类函数本来想重写，但参数写错，就会变成隐藏或新的成员函数。`override` 可以立刻暴露这个错误。

```cpp
struct B {
    virtual void f(int);
};

struct D : B {
    // void f(double);          // 不是重写，而是隐藏 B::f
    // void f(double) override; // 编译报错，帮助发现签名错误
};
```

#### 3. 重定义 hide/name hiding

重定义也常被称为隐藏，是指派生类声明了与基类同名的成员，导致基类中同名成员在派生类作用域中被隐藏。它不要求基类函数是虚函数，也不要求参数列表相同。只要名字相同，基类的同名重载集合都会被隐藏。

重定义的特点：

- 发生在基类和派生类之间。
- 基类函数可以是普通函数，也可以是虚函数。
- 派生类只要声明同名函数，就会隐藏基类同名函数集合。
- 若签名不匹配虚函数，则不会发生动态多态。
- 可以通过 `using Base::f;` 把基类重载集合引入派生类作用域。

```cpp
struct Base {
    void f() {}
    void f(int) {}
};

struct Derived : Base {
    void f(double) {} // 隐藏 Base::f() 和 Base::f(int)
};

Derived d;
// d.f();    // 错误：Base::f() 被隐藏
// d.f(1);   // 调用 Derived::f(double)，而不是 Base::f(int)
d.f(1.2);
```

如果希望派生类增加一个重载，同时保留基类重载，应写：

```cpp
struct Derived2 : Base {
    using Base::f;
    void f(double) {}
};

Derived2 d2;
d2.f();    // Base::f()
d2.f(1);   // Base::f(int)
d2.f(1.2); // Derived2::f(double)
```

#### 4. 三者对比

| 概念 | 英文 | 条件 | 绑定时机 | 示例 |
| --- | --- | --- | --- | --- |
| 重载 | overload | 同名，不同参数列表 | 编译期 | `f(int)` / `f(double)` |
| 重写 | override | 派生类覆盖基类虚函数 | 运行期 | `void f() override` |
| 重定义/隐藏 | hide | 派生类出现同名成员 | 编译期名字查找 | `Derived::f(int)` 隐藏 `Base::f()` |

```cpp
struct Base {
    virtual void f(int) {}
    void g() {}
    void h() {}
};
struct Derived : Base {
    void f(int) override {} // 重写
    void g(int) {}          // 重定义/隐藏 Base::g()
    void h() {}             // 重定义/隐藏 Base::h()
    void h(double) {}       // 在 Derived 内部构成重载
};
```

#### 5. 常见误区

- 仅返回值不同不能构成重载。
- 派生类函数名相同但参数不同，不是重写，而是隐藏。
- 基类函数不是 `virtual` 时，派生类写同名同参函数也不是动态多态意义上的重写，而是隐藏。
- 构造函数不能被继承体系中的普通方式重写，但可以重载。
- 析构函数可以是虚函数，派生类析构会覆盖基类虚析构入口，但析构函数名字看起来不同，这是语言特殊规则。

#### 常见追问

- **为什么建议写 `override`？**  
  因为它能让编译器检查派生类函数是否真的覆盖了基类虚函数，避免参数类型、`const`、引用限定等细节写错后悄悄变成隐藏。

- **重载是否一定发生在普通函数中？**  
  不是。成员函数、构造函数、运算符函数都可以重载。

- **重写和动态绑定必须满足什么调用形式？**  
  通常要通过基类指针或基类引用调用虚函数。如果发生对象切片，动态类型被切掉，就不会表现出派生类行为。

> [!TIP]
> 面试回答这题不要只背“重载同名不同参，重写虚函数，重定义隐藏”。更好的回答顺序是：先定义三者，再说发生位置和条件，再说明绑定时机，最后用 `override` 和 `using Base::f` 展示你知道工程中的规避方式。

### 10. 说一说 `strcpy`、`sprintf` 与 `memcpy` 的不同之处。

#### 面试回答

`strcpy` 复制以 `\0` 结尾的 C 字符串；`sprintf` 按格式化规则把内容写入字符缓冲区；`memcpy` 按指定字节数复制任意内存。三者都要求目标空间足够，否则可能导致缓冲区溢出。`memcpy` 不处理内存重叠，重叠时应使用 `memmove`。

#### 1. `strcpy`

`strcpy` 的原型通常是：

```cpp
char* strcpy(char* dest, const char* src);
```

它从 `src` 开始逐字符复制，直到遇到字符串结束符 `\0`，并且会把这个 `\0` 一起复制到 `dest`。它面向的是 C 风格字符串，不需要调用者传入复制长度。

特点：

- 复制对象必须是以 `\0` 结尾的字符串。
- 返回值是目标缓冲区 `dest`。
- 不知道 `dest` 的容量，无法防止溢出。
- 如果 `src` 没有 `\0`，会一直越界读。
- 如果源和目标内存重叠，行为未定义。

```cpp
char dst[8];
const char* src = "abc";
std::strcpy(dst, src); // dst 内容为 {'a','b','c','\0',...}
```

#### 2. `sprintf`

`sprintf` 的原型通常是：

```cpp
int sprintf(char* str, const char* format, ...);
```

它按照格式字符串把多个参数格式化后写入 `str`，返回写入字符数，不包含末尾 `\0`。它适合把数字、字符串等拼接成 C 字符串。

特点：

- 用途是格式化输出，而不是单纯复制。
- 会自动追加字符串结束符 `\0`。
- 不知道目标缓冲区大小，容易溢出。
- 应优先使用 `snprintf`，显式传入缓冲区大小。

```cpp
char buf[32];
std::snprintf(buf, sizeof(buf), "id=%d,name=%s", 7, "tom");
```

#### 3. `memcpy`

`memcpy` 的原型通常是：

```cpp
void* memcpy(void* dest, const void* src, size_t n);
```

它从 `src` 复制固定 `n` 个字节到 `dest`，不关心内容是否是字符串，也不关心是否有 `\0`。因此它可用于复制数组、结构体的原始字节、网络数据包等。

特点：

- 面向任意内存块，不要求内容是字符串。
- 复制长度由调用者指定。
- 不会自动补 `\0`。
- 源和目标重叠时行为未定义，重叠应使用 `memmove`。
- 对非平凡 C++ 对象不能随便 `memcpy` 复制，否则会破坏对象语义。

| 函数 | 处理对象 | 结束条件 | 主要风险 | 推荐替代 |
| --- | --- | --- | --- | --- |
| `strcpy(dest, src)` | C 字符串 | 遇到 `\0` | 不知道 dest 大小 | `strncpy` 也有坑，更推荐 `std::string` 或 `snprintf` |
| `sprintf(buf, fmt, ...)` | 格式化字符串 | 格式化结束 | 缓冲区溢出 | `snprintf` |
| `memcpy(dest, src, n)` | 任意字节 | 固定 n 字节 | 重叠 UB、越界 | `memmove` 用于重叠 |

#### 常见追问

- **`memcpy` 和 `memmove` 的区别？**  
  `memcpy` 要求源和目标不重叠；`memmove` 会处理重叠，通常内部会根据地址方向选择从前往后或从后往前复制。

- **`strncpy` 是否一定安全？**  
  不一定。`strncpy` 如果源字符串长度达到或超过指定长度，目标缓冲区可能不会以 `\0` 结尾；如果源很短，它还会用大量 `\0` 填充，性能也可能不好。

- **C++ 中更推荐什么？**  
  普通字符串处理优先用 `std::string`；格式化可用 `std::format`（C++20）或 `snprintf`；二进制内存复制只用于平凡可复制类型。

### 11. `strlen` 和 `sizeof` 区别。

#### 面试回答

`strlen` 是库函数，运行时从给定地址开始数到第一个 `\0`，返回字符串长度，不包含 `\0`。`sizeof` 是运算符，编译期多数情况下即可确定，返回类型或对象占用的字节数。数组在本作用域用 `sizeof` 可得到整个数组大小，但作为函数参数会退化为指针。

#### 1. `strlen`

`strlen` 的原型通常是：

```cpp
size_t strlen(const char* s);
```

它从 `s` 指向的位置开始向后扫描，直到遇到第一个 `\0`，返回扫描到的普通字符数量。它只适用于 C 风格字符串。

特点：

- 运行时计算，时间复杂度 O(n)。
- 不包含末尾 `\0`。
- 参数必须是合法指针，并且后续内存中必须存在 `\0`。
- 对未初始化字符数组调用 `strlen` 是危险的。

#### 2. `sizeof`

`sizeof` 是 C/C++ 运算符，用于求类型或表达式结果类型所占字节数。大部分情况下编译期即可确定，返回类型是 `size_t`。

特点：

- 求的是内存占用字节数，不是字符串长度。
- 对数组名可得到整个数组大小。
- 对指针只得到指针自身大小。
- 对表达式通常不求值。
- 对类对象会考虑成员、虚指针和对齐填充。

```cpp
char a[] = "abc";
const char* p = "abc";

sizeof(a);  // 4，包含 '\0'
strlen(a);  // 3
sizeof(p);  // 指针大小，64 位通常 8
strlen(p);  // 3
```

函数传参时数组会退化：

```cpp
void f(char arr[]) {
    sizeof(arr); // 指针大小，不是数组大小
}
```

#### 对比表

| 对比项 | `strlen` | `sizeof` |
| --- | --- | --- |
| 类型 | 库函数 | 运算符 |
| 计算内容 | 字符串有效字符数 | 类型/对象占用字节数 |
| 是否包含 `\0` | 不包含 | 如果是字符数组，会包含存储的 `\0` |
| 计算时机 | 运行时 | 多数为编译期 |
| 参数要求 | 必须是以 `\0` 结尾的字符序列 | 类型或表达式 |
| 数组传参 | 不关心数组原始大小 | 形参中数组退化为指针 |

> [!CAUTION]
> `strlen` 要求参数指向合法且以 `\0` 结尾的字符数组，否则会越界读取。`sizeof` 对表达式通常不求值，如 `sizeof(i++)` 不会让 `i` 增加。

### 12. 二维数组是什么，函数指针是什么？

#### 面试回答

二维数组是“数组的数组”，例如 `int a[3][4]` 表示 3 个元素，每个元素是 `int[4]`，整体内存连续，按行优先存储。函数指针是保存函数入口地址的指针，类型由返回值和参数列表决定，可以用于回调、策略选择和表驱动分发。

```cpp
int a[3][4]{};
int (*row)[4] = a;       // 指向一行 4 个 int 的数组指针
int x = a[1][2];         // 等价于 *(*(a + 1) + 2)

int add(int x, int y) { return x + y; }
int (*fp)(int, int) = add;
int r = fp(1, 2);
```

> [!CAUTION]
> 函数指针 `int (*fp)(int)` 和返回指针的函数 `int* fp(int)` 完全不同，括号决定 `*` 和名字的结合方式。

### 13. 简述值传递、指针传递、地址传递的区别。

#### 面试回答

值传递会复制实参，函数内修改形参不影响实参；指针传递复制的是地址，函数可以通过解引用修改实参，也能表达“可能为空”；引用传递是对象别名，语义上更像直接操作原对象，通常不能重新绑定。大对象只读传参常用 `const T&`，需要转移资源时用右值引用 `T&&`。

```cpp
void byValue(int x) { x = 10; }
void byPointer(int* p) { if (p) *p = 10; }
void byRef(int& r) { r = 10; }
```

#### 1. 值传递

值传递是把实参的值复制一份给形参。函数内部操作的是副本，除非对象内部共享了某些外部资源，否则不会影响原对象。

特点：

- 形参和实参是两个对象。
- 修改形参不影响实参。
- 对内置类型和小对象成本低。
- 对大对象可能有拷贝开销。
- 若类型禁用了拷贝，则不能按值传递。

```cpp
void f(int x) {
    x = 100;
}

int a = 1;
f(a); // a 仍然是 1
```

#### 2. 指针传递，也叫地址传递

指针传递是把对象地址作为参数传入函数。指针本身仍然是按值复制，但复制的是地址，所以函数可以通过该地址访问或修改原对象。

特点：

- 形参指针是实参地址的一份副本。
- 可以通过 `*p` 修改原对象。
- 可以传 `nullptr`，因此函数内部通常要判空。
- 如果希望函数修改调用者的指针本身，需要传二级指针或指针引用。

```cpp
void setValue(int* p) {
    if (p) *p = 100;
}

void reset(int*& p) {
    delete p;
    p = nullptr;
}
```

#### 3. 引用传递

引用传递是给原对象起别名。函数内部使用引用形参，就像直接使用实参对象。

特点：

- 定义时必须绑定对象。
- 一般不能重新绑定。
- 语法更简洁，不需要解引用。
- 语义上表示参数不能为空。
- `const T&` 是只读引用，常用于避免大对象拷贝。

```cpp
void setValue(int& r) {
    r = 100;
}

void print(const std::string& s) {
    std::cout << s << "\n";
}
```

#### 对比表

| 对比项 | 值传递 | 指针传递 | 引用传递 |
| --- | --- | --- | --- |
| 是否复制对象 | 复制对象值 | 复制地址 | 不复制对象 |
| 能否修改原对象 | 通常不能 | 可以 | 可以 |
| 能否表示空 | 不涉及 | 可以为 `nullptr` | 通常不为空 |
| 语法 | `x` | `*p`、`p->` | `r` |
| 适用场景 | 小对象、需要副本 | 可选对象、C 接口、改变指向 | 必选对象、输出参数、大对象只读 |

#### 选择建议

- 小型内置类型：按值传递。
- 大对象只读：`const T&`。
- 必须修改且不能为空：`T&`。
- 可以为空或需要改变指向：`T*` 或 `T**`。
- 表达所有权转移：`std::unique_ptr<T>` 或 `T&&`。

#### 常见追问

- **指针传递是不是引用传递？**  
  严格说不是。指针传递时“指针变量本身”仍然是按值传递，只是这个值是地址。通过地址可以修改原对象，但不能直接改变调用者的指针变量，除非传 `T**` 或 `T*&`。

- **为什么大对象常用 `const T&`？**  
  因为它避免拷贝，又承诺不修改对象，是 C++ 中表达只读输入参数的常见方式。

### 14. C++ 中 `const` 关键字的作用？

#### 面试回答

`const` 表示“不能通过该名字修改对象”，可修饰变量、指针、引用、函数参数、返回值、成员函数和成员变量。它的核心价值是表达只读语义、让编译器检查误修改、提升接口可读性。类中 `const` 成员必须在初始化列表中初始化，`const` 成员函数承诺不修改对象的逻辑状态。

#### 常见用法

```cpp
const int a = 10;       // 常量对象
const int* p1 = &a;     // 指向 const int 的指针，不能通过 p1 改值
int* const p2 = nullptr;// const 指针，p2 自身不能改指向
const int* const p3 = &a;

void print(const std::string& s); // 不修改参数，避免拷贝

class X {
public:
    int value() const { return value_; } // const 成员函数
private:
    int value_{};
};
```

#### 1. 修饰普通变量

```cpp
const int maxSize = 1024;
```

变量初始化后不能再通过该变量名修改。对于内置类型，如果初始化表达式是编译期常量，`const` 变量还可以用于数组大小、模板参数等编译期场景。

#### 2. 修饰指针

判断指针相关 `const` 时，可以看 `const` 离谁近：

```cpp
const int* p;       // p 指向的 int 不能通过 p 修改
int const* p2;      // 与 const int* 等价
int* const p3 = &x; // p3 这个指针不能改指向
const int* const p4 = &x; // 指针和指向内容都不能通过 p4 改
```

#### 3. 修饰函数参数和返回值

函数参数使用 `const T&` 能避免拷贝并承诺不修改实参：

```cpp
void print(const std::string& s);
```

返回 `const` 值在现代 C++ 中通常意义不大，还可能影响移动语义；但返回 `const` 引用可用于暴露只读访问。

#### 4. 修饰成员函数

成员函数后面的 `const` 修饰的是隐式 `this` 指针，表示该函数不能修改对象的逻辑状态。

```cpp
class Person {
public:
    const std::string& name() const { return name_; }
    void rename(std::string name) { name_ = std::move(name); }
private:
    std::string name_;
};
```

`const` 对象只能调用 `const` 成员函数：

```cpp
const Person p;
p.name();   // 正确
// p.rename("x"); // 错误
```

#### 易错点

- `const int*` 和 `int const*` 等价，都是“指向常量的指针”。
- `int* const` 是“常量指针”，指针本身不能改。
- `const` 对象只能调用 `const` 成员函数。
- `mutable` 成员可在 `const` 成员函数中修改，常用于缓存、引用计数、互斥锁。
- `const_cast` 只能去掉表达式层面的 const 属性，如果对象本身真的是 const，去掉后修改仍是未定义行为。

#### 常见追问

- **`const` 和宏常量有什么区别？**  
  `const` 受类型系统和作用域约束，有调试信息；宏只是文本替换。

- **`const` 和 `constexpr` 区别？**  
  `const` 表示只读，未必是编译期常量；`constexpr` 要求可用于常量表达式语境，强调编译期求值能力。

### 15. C++ 中 `static` 关键字的作用？

#### 面试回答

`static` 的作用取决于上下文：局部静态变量具有静态存储期，只初始化一次；文件作用域的 `static` 变量或函数具有内部链接，只在当前翻译单元可见；类静态数据成员属于类而不是对象；静态成员函数没有 `this` 指针，只能直接访问静态成员。

| 场景 | 含义 | 示例 |
| --- | --- | --- |
| 函数内局部变量 | 静态存储期，首次执行到定义处初始化 | `static int cnt;` |
| 文件作用域变量/函数 | 内部链接，限制符号可见性 | `static void helper();` |
| 类静态数据成员 | 所有对象共享一份 | `static int count;` |
| 类静态成员函数 | 属于类，无 `this` | `static void f();` |

```cpp
int nextId() {
    static int id = 0; // C++11 起初始化线程安全
    return ++id;
}
```

#### 1. 局部静态变量

局部静态变量定义在函数内部，但生命周期不是函数调用期间，而是整个程序运行期间。它只在第一次执行到定义语句时初始化一次，之后再次进入函数会保留上次的值。

```cpp
int countCall() {
    static int count = 0;
    return ++count;
}
```

特点：

- 作用域仍然是函数内部，函数外不可见。
- 生命周期是整个程序。
- C++11 起初始化线程安全。
- 常用于懒加载单例、计数器、缓存等。

#### 2. 文件作用域 `static`

在全局变量或普通函数前加 `static`，表示内部链接，只在当前 `.cpp` 文件可见。

```cpp
// file1.cpp
static int g_count = 0;
static void helper() {}
```

这可以避免符号污染和跨文件命名冲突。现代 C++ 中也常用匿名命名空间达到类似效果。

#### 3. 类静态数据成员

静态数据成员属于类，不属于某个对象，所有对象共享一份。

```cpp
class A {
public:
    static int count;
};

int A::count = 0; // C++17 前通常需要类外定义
```

C++17 起可以使用 `inline static` 在类内定义：

```cpp
class B {
public:
    inline static int count = 0;
};
```

#### 4. 类静态成员函数

静态成员函数属于类，没有 `this` 指针，因此不能直接访问非静态成员。

```cpp
class A {
public:
    static void printCount() {
        std::cout << count << "\n";
        // std::cout << value_; // 错误：没有 this
    }
private:
    static int count;
    int value_{};
};
```

#### 常见追问

- **局部静态变量存在哪里？**  
  存在静态存储区，而不是栈上；只是作用域限制在函数内。

- **`static` 全局变量和普通全局变量区别？**  
  生命周期相同，都是整个程序；链接属性不同，`static` 全局变量只在当前翻译单元可见。

- **静态成员函数能不能是虚函数？**  
  不能。虚函数依赖对象和 `this` 指针进行动态分派，静态成员函数没有 `this`。

> [!CAUTION]
> 静态成员函数不能直接访问非静态成员，因为没有具体对象，也没有 `this` 指针。

### 16. C++ 中 `class` 和 `struct` 的区别？

#### 面试回答

C++ 中 `class` 和 `struct` 的能力基本相同，都可以有成员函数、构造析构、继承、模板、访问控制和虚函数。主要区别是默认访问权限和默认继承权限：`class` 默认 `private`，`struct` 默认 `public`。工程习惯上，`struct` 常用于简单数据聚合，`class` 常用于需要封装不变量和行为的类型。

```cpp
struct S { int x; };       // x 默认 public
class C { int x; };        // x 默认 private

struct D1 : S {};          // 默认 public 继承
class D2 : S {};           // 默认 private 继承
```

#### 详细说明

在 C++ 中，`struct` 并不是 C 语言结构体的简单延续，它已经具备面向对象能力：

- 可以定义构造函数、析构函数。
- 可以定义普通成员函数和静态成员函数。
- 可以使用 `public/private/protected`。
- 可以继承，也可以被继承。
- 可以有虚函数和虚析构函数。
- 可以作为模板参数或定义类模板。

两者真正的语言级差异主要是默认权限：

| 对比项 | `struct` | `class` |
| --- | --- | --- |
| 成员默认访问权限 | `public` | `private` |
| 继承默认访问权限 | `public` | `private` |
| 是否能有构造/析构 | 能 | 能 |
| 是否能有虚函数 | 能 | 能 |
| 是否能继承 | 能 | 能 |

工程习惯：

- 用 `struct` 表示纯数据、配置项、消息结构、返回多个值的简单聚合。
- 用 `class` 表示有封装、有不变量、有行为约束的对象。

> [!CAUTION]
> 不要回答“C++ 的 struct 不能有函数”。这是 C 语言结构体的特点，不是 C++ 的特点。

### 17. `string` 的 SSO 模式。

#### 面试回答

SSO 是 Small String Optimization，短字符串优化。`std::string` 对象内部通常预留一小段缓冲区，当字符串较短时直接存在对象内部，不进行堆分配；当长度超过阈值后才在堆上分配。它能显著减少短字符串创建、拷贝和析构时的动态内存开销。

#### 详细解释

一个 `std::string` 通常需要保存长度、容量和数据指针。SSO 实现会把“小缓冲区”和“指针/容量字段”复用在同一对象空间内。短字符串直接放在对象里；长字符串则让指针指向堆内存。

```cpp
std::string a = "abc";        // 大概率使用 SSO，不分配堆
std::string b(1000, 'x');     // 通常需要堆分配
```

> [!CAUTION]
> SSO 是标准库实现优化，不是 C++ 标准规定的具体阈值。不同 libstdc++、libc++、MSVC STL 的阈值可能不同，不要写依赖阈值的代码。

### 18. C++ 中类型转换有哪几种？简述区别。

#### 面试回答

C++ 推荐使用四种显式类型转换：`static_cast`、`dynamic_cast`、`const_cast`、`reinterpret_cast`。它们比 C 风格强转更清晰，能表达转换意图，也便于编译器和代码审查发现风险。

| 转换 | 主要用途 | 运行期检查 | 风险 |
| --- | --- | --- | --- |
| `static_cast` | 基本类型转换、上行转换、明确的非多态转换 | 无 | 下行转换不安全 |
| `dynamic_cast` | 多态类型安全下行/交叉转换 | 有 | 需要至少一个虚函数，有运行期开销 |
| `const_cast` | 添加/移除 const/volatile | 无 | 修改本来为 const 的对象是未定义行为 |
| `reinterpret_cast` | 按位重新解释、底层地址转换 | 无 | 极不安全，可能破坏别名、对齐和可移植性 |

```cpp
double d = 3.14;
int i = static_cast<int>(d);

Base* b = new Derived;
if (auto* p = dynamic_cast<Derived*>(b)) {
    // p 安全可用
}
```

#### 1. `static_cast`

`static_cast` 用于相对“正常”的类型转换，编译器在编译期完成转换规则检查，但不做运行期安全检查。

常见用途：

- 基本算术类型转换，如 `double` 转 `int`。
- `void*` 和具体对象指针之间转换。
- 基类指针和派生类指针之间的上行转换；下行转换也能写，但不安全。
- 调用显式转换构造函数或类型转换运算符。

```cpp
double d = 3.9;
int i = static_cast<int>(d); // i = 3，截断

Derived* pd = new Derived;
Base* pb = static_cast<Base*>(pd); // 上行转换安全
```

#### 2. `dynamic_cast`

`dynamic_cast` 主要用于多态类型的安全下行转换或交叉转换。它要求源类型至少有一个虚函数，因为运行期需要 RTTI 信息判断对象真实类型。

```cpp
Base* b = getObject();
if (Derived* d = dynamic_cast<Derived*>(b)) {
    d->onlyDerived();
} else {
    // b 实际上不是 Derived
}
```

特点：

- 指针转换失败返回 `nullptr`。
- 引用转换失败抛 `std::bad_cast`。
- 有运行期开销。
- 适合无法通过设计避免的运行期类型判断，但滥用会说明多态接口可能设计得不够好。

#### 3. `const_cast`

`const_cast` 用于增加或移除 `const/volatile` 限定。它只改变表达式类型，不改变对象本身是否真的是常量。

```cpp
void legacy_api(char* p);

void f(const char* s) {
    legacy_api(const_cast<char*>(s)); // 只有 legacy_api 不修改内容时才勉强安全
}
```

如果对象本身定义为 `const`，去掉 const 后修改它是未定义行为：

```cpp
const int x = 1;
int* p = const_cast<int*>(&x);
// *p = 2; // 未定义行为
```

#### 4. `reinterpret_cast`

`reinterpret_cast` 是最底层、最危险的转换，表示把一段位模式按另一种类型解释。它常见于系统编程、硬件寄存器、序列化、网络协议解析等低层场景。

风险：

- 可能违反严格别名规则。
- 可能产生未对齐访问。
- 结果强依赖平台、ABI、端序。
- 不会做语义转换，只是改变编译器解释方式。

```cpp
std::uintptr_t addr = reinterpret_cast<std::uintptr_t>(pb);
```

#### C 风格强转的问题

C 风格强转 `(T)x` 可能依次尝试多种 C++ 转换，具体意图不清楚，容易把 `const_cast`、`static_cast`、`reinterpret_cast` 混在一起。工程中应优先使用 C++ 四种显式转换，让风险暴露在代码表面。

#### 常见追问

- **向上转型和向下转型分别用什么？**  
  派生类转基类是向上转型，天然安全，普通隐式转换即可；基类转派生类是向下转型，如果需要运行期安全检查，用 `dynamic_cast`。

- **为什么 `dynamic_cast` 要求多态类型？**  
  因为它要根据 RTTI 判断对象真实类型，而 RTTI 通常通过含虚函数类的虚表体系关联。

- **`reinterpret_cast` 能不能用于网络字节序转换？**  
  不能替代字节序处理。网络字节序转换应使用 `htonl/ntohl` 等函数或显式按字节解析。

> [!TIP]
> 面试可补充：能不用强转就不用；能用 `static_cast` 就不要用 C 风格强转；`reinterpret_cast` 通常只在底层、序列化、系统编程中谨慎使用。

### 19. C++ 中函数指针和指针函数的区别？

#### 面试回答

函数指针是指向函数的指针，保存函数入口地址；指针函数是返回指针的函数。判断方式是看 `*` 和名字如何结合，括号会改变优先级。

```cpp
int* f1(int);        // 指针函数：返回 int*
int (*f2)(int);      // 函数指针：指向 int(int)
```

函数指针常用于回调、插件表、状态机跳转表。C++ 中更通用的可调用对象包装是 `std::function`，也可以使用 lambda、函数对象和成员函数指针。

#### 函数指针

函数指针的本质是“指向函数入口地址的指针”。函数编译后位于代码段，函数名在表达式中可退化为函数地址。

```cpp
int add(int a, int b) {
    return a + b;
}

int (*fp)(int, int) = add;
int ret = fp(1, 2);
```

函数指针类型必须与目标函数签名匹配，包括返回值和参数类型。

#### 指针函数

指针函数是“返回指针的函数”，重点是它先是一个函数，只是返回值是指针。

```cpp
int global = 10;

int* getPtr() {
    return &global;
}
```

#### 如何读复杂声明？

可以从变量名开始，结合运算符优先级读：

```cpp
int (*fp)(int); // fp 先和 * 结合，所以 fp 是指针；再看右边 (int)，指向函数
int* f(int);    // f 先和 (int) 结合，所以 f 是函数；返回 int*
```

> [!TIP]
> 面试中如果担心复杂声明读错，可以主动说“函数指针要用括号把 `*fp` 括起来，否则就变成返回指针的函数”。

### 20. C++ 中有哪些智能指针，它们有什么异同？

#### 面试回答

标准智能指针主要有 `std::unique_ptr`、`std::shared_ptr`、`std::weak_ptr`。`unique_ptr` 表示独占所有权，不能拷贝但可移动；`shared_ptr` 用引用计数表示共享所有权；`weak_ptr` 不拥有对象，只观察 `shared_ptr` 管理的对象，用于打破循环引用。它们都体现 RAII：对象生命周期结束时自动释放资源。

| 智能指针 | 所有权 | 拷贝 | 典型场景 |
| --- | --- | --- | --- |
| `unique_ptr` | 独占 | 禁止，可移动 | 工厂返回对象、类成员资源 |
| `shared_ptr` | 共享 | 允许，增加引用计数 | 多模块共享生命周期 |
| `weak_ptr` | 不拥有 | 允许 | 缓存、观察者、打破环 |

```cpp
auto p = std::make_unique<int>(1);
auto sp = std::make_shared<std::string>("hello");
std::weak_ptr<std::string> wp = sp;
```

#### 1. `unique_ptr`

`unique_ptr` 表示唯一所有权，同一时刻只能有一个 `unique_ptr` 负责释放资源。它不能拷贝，只能移动。

```cpp
std::unique_ptr<int> p1 = std::make_unique<int>(10);
// std::unique_ptr<int> p2 = p1;        // 错误：不能拷贝
std::unique_ptr<int> p2 = std::move(p1); // 正确：所有权转移
```

特点：

- 几乎没有额外运行期开销，大小通常等于一个裸指针。
- 适合表达独占资源，如文件句柄包装、树节点所有权、工厂函数返回值。
- 可配置自定义删除器，管理非 `new` 资源。

#### 2. `shared_ptr`

`shared_ptr` 表示共享所有权，多个 `shared_ptr` 可以共同管理同一个对象。最后一个强引用销毁或 reset 时，对象被删除。

它通常包含两部分：

- 指向对象的指针。
- 指向控制块的指针。控制块中保存强引用计数、弱引用计数、删除器、分配器等。

```cpp
auto sp1 = std::make_shared<int>(10);
auto sp2 = sp1; // 强引用计数 +1
```

特点：

- 适合生命周期确实需要共享的对象。
- 引用计数增减通常是原子操作，有一定开销。
- 控制块线程安全，但对象本身不自动线程安全。
- 容易因循环引用导致内存泄漏。

#### 3. `weak_ptr`

`weak_ptr` 是弱引用，不拥有对象，不增加强引用计数。它用于观察 `shared_ptr` 管理的对象是否还活着，也用于打破循环引用。

```cpp
std::weak_ptr<int> wp = sp1;
if (auto locked = wp.lock()) {
    std::cout << *locked << "\n";
}
```

特点：

- 不能直接解引用。
- 使用前要调用 `lock()` 得到临时 `shared_ptr`。
- 对象已释放时，`lock()` 返回空。

#### 常见追问

- **为什么推荐 `make_shared`？**  
  它通常一次分配对象和控制块，减少内存分配次数，并且异常安全更好。

- **为什么推荐 `make_unique`？**  
  写法简洁，避免裸 `new`，异常安全，所有权表达明确。

- **智能指针能完全替代裸指针吗？**  
  不能。智能指针表达所有权；裸指针和引用仍适合表达非拥有的临时观察关系。

- **什么时候不要用 `shared_ptr`？**  
  如果所有权清晰且唯一，应优先用 `unique_ptr`。滥用 `shared_ptr` 会让生命周期关系模糊，并增加引用计数开销。

### 21. `void*` 的大小是多少？

#### 面试回答

`void*` 是对象指针，大小等于当前进程的指针宽度。32 位进程通常是 4 字节，64 位进程通常是 8 字节。它与指向什么类型无关，`int*`、`char*`、`void*` 在同一平台通常大小相同。

#### 注意点

`void*` 可以保存任意对象地址，但不能直接解引用，因为编译器不知道对象大小和解释方式。使用前必须转换为具体类型指针。

```cpp
int x = 1;
void* p = &x;
int* q = static_cast<int*>(p);
```

> [!CAUTION]
> 函数指针不一定能安全转换成 `void*`，C++ 标准对对象指针和函数指针是分开的。

### 22. 简述 `malloc/free` 的实现原理。为何 `free(p)` 只需要传堆空间地址？

#### 面试回答

`malloc` 本质上是用户态内存分配器，从操作系统申请大块内存后，在用户态维护空闲块和已分配块。分配时查找合适大小的空闲块，必要时切分；释放时根据块元数据标记为空闲，并尝试与相邻空闲块合并。`free(p)` 只需要用户区地址，是因为分配器通常在返回地址前后保存了块大小、状态、链表指针等元数据。

#### 典型流程

1. 对请求大小做对齐，并加上元数据开销。
2. 在不同大小的 bin/free list 中找合适空闲块。
3. 找不到时通过 `brk/sbrk` 或 `mmap` 向操作系统申请更多内存。
4. 若空闲块太大，切分为已分配块和剩余空闲块。
5. `free` 时通过 `p` 回退找到 chunk header，更新状态并合并。

```text
[ chunk header | 用户可用内存 .... ]
               ^
               malloc 返回给用户的 p
```

#### 1. `malloc` 如何向操作系统要内存？

分配器一般不会每次 `malloc` 都直接调用系统调用，因为系统调用成本较高。它通常先从操作系统申请一大块区域，再在用户态自己管理。

常见方式：

- 小块或连续堆扩展：传统上可通过 `brk/sbrk` 扩展 program break。
- 大块分配：常通过 `mmap` 单独映射，释放时可直接 `munmap` 归还系统。
- 多线程场景：分配器可能使用多个 arena 或线程缓存，减少锁竞争。

#### 2. 空闲块如何管理？

分配器通常按大小把空闲块分到不同链表或桶中，例如 small bin、large bin、fast bin、tcache 等。申请内存时先找合适大小的空闲块；如果空闲块过大，就切出一部分给用户，剩余部分继续作为空闲块管理。

释放时可能发生：

- 把块标记为空闲。
- 放回对应大小的空闲链表。
- 和相邻空闲块合并，减少外部碎片。
- 在满足条件时把大块内存归还操作系统。

#### 3. 为什么 `free` 不需要传大小？

因为分配器在内部保存了大小信息。用户拿到的指针通常指向“用户可用区域”，而在它前面或旁边有分配器元数据。`free(p)` 可以通过 `p` 找回元数据，知道这块内存多大、是否空闲、相邻块在哪里。

```text
低地址
+----------------+-------------------------+
| size/status... | user bytes returned p   |
+----------------+-------------------------+
                 ^
                 p
高地址
```

#### 4. 相关风险

- `free` 非 malloc 家族返回的指针：未定义行为。
- `free(p + 1)`：不是原始返回地址，未定义行为。
- double free：破坏分配器元数据，可能崩溃或形成安全漏洞。
- use-after-free：释放后继续读写，可能破坏后续分配对象。
- 堆越界写：可能覆盖 chunk header，使后续 `free` 崩溃。

> [!CAUTION]
> `free` 后指针变量本身的值不会自动变空，继续使用就是悬空指针。重复 `free` 同一地址也是未定义行为。

### 23. `malloc` 申请内存后，怎么保证一定申请到了？会直接使用吗？

#### 面试回答

`malloc` 不保证一定成功，失败会返回 `NULL`，所以使用前必须检查返回值。申请成功后也不能越界写入，还要根据需要初始化，因为 `malloc` 得到的是未初始化的原始内存。如果需要清零可用 `calloc` 或 `memset`，但对象类型在 C++ 中应优先用 `new`、容器或智能指针。

```c
size_t n = 100;
int* p = (int*)malloc(n * sizeof(int));
if (p == NULL) {
    perror("malloc");
    return;
}
memset(p, 0, n * sizeof(int));
free(p);
```

#### 补充

Linux 有 overcommit 机制时，`malloc` 返回非空也不等于物理内存已经全部到位，真正访问页面时可能触发缺页并失败。但从普通应用层面，必须先做空指针检查，再正确初始化和释放。

### 24. `new/delete` 与 `malloc/free` 的异同。

#### 面试回答

它们都可用于动态内存管理，但层次不同。`malloc/free` 是 C 标准库函数，只分配和释放原始字节，不调用构造/析构；`new/delete` 是 C++ 运算符，会分配内存并构造对象，释放时先析构再归还内存。`new` 返回目标类型指针，失败默认抛 `std::bad_alloc`；`malloc` 返回 `void*`，失败返回 `NULL`。

| 对比项 | `new/delete` | `malloc/free` |
| --- | --- | --- |
| 类型 | C++ 运算符 | C 库函数 |
| 单位 | 类型对象 | 字节数 |
| 构造析构 | 会调用 | 不会调用 |
| 返回类型 | `T*` | `void*` |
| 失败行为 | 默认抛异常 | 返回 `NULL` |
| 重载/定制 | 可重载 `operator new` | 依赖分配器实现 |

#### `new` 的执行过程

表达式 `new T(args)` 大致分两步：

1. 调用 `operator new(sizeof(T))` 分配一块满足 `T` 对齐要求的原始内存。
2. 在这块内存上调用 `T` 的构造函数。

如果构造函数抛异常，运行时会自动调用匹配的 `operator delete` 释放刚才分配的原始内存，避免泄漏。

```cpp
T* p = new T(args);
```

可以近似理解为：

```cpp
void* raw = operator new(sizeof(T));
try {
    T* p = new (raw) T(args); // placement new 构造对象
} catch (...) {
    operator delete(raw);
    throw;
}
```

#### `delete` 的执行过程

表达式 `delete p` 也大致分两步：

1. 如果 `p` 非空，调用对象析构函数。
2. 调用 `operator delete` 释放原始内存。

因此 `delete nullptr;` 是安全的，什么也不做。

#### `malloc/free` 的边界

`malloc` 只返回一块未初始化的字节区域，不知道里面将来放什么类型对象；`free` 也只释放这块字节区域，不会调用析构函数。因此在 C++ 中用 `malloc` 创建非平凡对象是不完整的，必须手动 placement new 构造并手动析构。

```cpp
void* raw = std::malloc(sizeof(std::string));
auto* s = new (raw) std::string("abc");
s->~basic_string();
std::free(raw);
```

#### 常见追问

- **`new` 能不能重载？**  
  可以重载类专属或全局 `operator new/delete`，常用于内存池、调试分配器、对齐分配。

- **`malloc` 分配的内存是否满足对齐？**  
  标准 `malloc` 返回的内存满足任意基本对象类型的对齐要求；过对齐类型需要 C11 `aligned_alloc` 或 C++17 对齐版本 `operator new` 等机制。

- **为什么 C++ 中不建议直接用 `malloc/free` 管理对象？**  
  因为它绕过构造/析构，容易破坏对象生命周期。现代 C++ 应优先使用容器、智能指针和 RAII。

> [!CAUTION]
> 不能混用：`new` 出来的对象必须 `delete`，`malloc` 出来的内存必须 `free`。

### 25. `delete` 与 `delete[]`。

#### 面试回答

`delete` 用于释放单个对象，调用一次析构函数；`delete[]` 用于释放数组对象，对每个元素调用析构函数。`new[]` 必须配 `delete[]`，`new` 必须配 `delete`，混用是未定义行为。

```cpp
auto* p = new std::string("x");
delete p;

auto* arr = new std::string[10];
delete[] arr;
```

#### 原理说明

数组版本通常需要知道数组元素个数，以便逐个析构。实现可能在返回给用户的指针前保存元素数量或其他 cookie。若用错释放方式，析构次数、释放地址和分配器元数据都可能出错。

#### 详细区别

`delete p` 的语义是：

1. 调用 `p` 指向对象的析构函数一次。
2. 调用匹配的 `operator delete` 释放内存。

`delete[] p` 的语义是：

1. 对数组中每个已经构造的元素调用析构函数。
2. 调用匹配的 `operator delete[]` 释放整块数组内存。

```cpp
struct A {
    ~A() { std::cout << "~A\n"; }
};

A* p = new A;
delete p;       // 打印一次

A* arr = new A[3];
delete[] arr;   // 打印三次
```

#### 常见误区

- `new[]` 配 `delete` 是未定义行为，不只是“少调用几次析构”这么简单。
- 内置类型数组混用有时看起来没崩，但仍然是未定义行为。
- `delete nullptr` 和 `delete[] nullptr` 都是安全的。
- 如果类管理资源，数组释放方式错误更容易暴露为资源泄漏。

### 26. 指针和引用的区别。

#### 面试回答

指针是一个变量，保存地址，可以为空、可以改变指向、可以做指针运算；引用是对象别名，定义时必须初始化，之后不能重新绑定，语义上通常不为空。函数接口中，引用适合表达“必须有对象”，指针适合表达“可能为空”或“需要改变指向”。

| 对比 | 指针 | 引用 |
| --- | --- | --- |
| 是否可为空 | 可以为 `nullptr` | 正常语义下不可为空 |
| 是否必须初始化 | 不一定 | 必须 |
| 是否可重新绑定 | 可以 | 不可以 |
| 是否可运算 | 可 `++`、偏移 | 不可 |
| 使用语法 | `*p` / `p->` | 像普通对象 |

```cpp
void reset(int*& p); // 可以改变调用者指针本身
void inc(int& x);    // 修改调用者对象
```

#### 1. 本质语义

指针保存的是一个地址，引用是某个已有对象的别名。指针本身有自己的存储空间，可以被赋值为不同地址；引用一旦初始化绑定到对象，之后对引用的操作就是对原对象操作，不能再改绑到另一个对象。

```cpp
int a = 1, b = 2;
int* p = &a;
p = &b;   // 指针可以改指向

int& r = a;
r = b;    // 不是让 r 绑定 b，而是把 b 的值赋给 a
```

#### 2. 空值语义

指针可以自然表示“没有对象”：

```cpp
void f(int* p) {
    if (!p) return;
}
```

引用通常表达“必须有对象”：

```cpp
void g(int& x) {
    ++x;
}
```

虽然可以通过解引用空指针制造“空引用”，但那本身已经是未定义行为，不能作为合法语义使用。

#### 3. 作为函数参数的设计差异

- 参数一定存在并且要修改：用 `T&`。
- 参数一定存在但只读：用 `const T&`。
- 参数可选：用 `T*`，或现代 C++ 中用 `std::optional<std::reference_wrapper<T>>` 表达。
- 需要改变调用者指针指向：用 `T*&` 或 `T**`。
- 表达资源所有权：用智能指针，不要只靠裸指针。

#### 4. 常见追问

- **引用是否占内存？**  
  语言语义上引用是别名，不是对象；实现上编译器可能用指针实现引用参数或引用成员。因此不能简单说“引用一定不占内存”，要区分语言语义和实现。

- **引用和指针哪个更安全？**  
  引用语义更强，避免了空值和改绑，但不能表达可选对象；指针更灵活，但需要判空和生命周期管理。

- **引用成员有什么要求？**  
  引用成员必须在构造函数初始化列表中初始化，不能在构造函数体内再绑定。

### 27. 引用作为函数返回时为什么不能返回局部变量？

#### 面试回答

局部变量位于函数栈帧中，函数返回后生命周期结束，引用它会产生悬空引用。后续访问悬空引用是未定义行为，可能看似正常，也可能崩溃或读到被覆盖的数据。正确做法是按值返回、返回静态对象引用、返回外部传入对象引用，或返回堆对象的智能指针。

```cpp
int& bad() {
    int x = 1;
    return x; // 错误：返回局部变量引用
}

int good() {
    int x = 1;
    return x; // 正确：返回值优化/移动
}
```

#### 生命周期分析

局部自动变量的生命周期从定义处开始，到所在作用域结束。函数返回时，当前函数栈帧被销毁，其中的局部变量也随之失效。引用本身不会延长普通局部变量生命周期，因此返回局部变量引用会让调用者拿到一个已经失效对象的别名。

```cpp
const std::string& badString() {
    std::string s = "hello";
    return s; // s 已经销毁
}
```

#### 哪些返回引用是合理的？

- 返回全局对象或静态对象引用，但要注意线程安全和全局状态。
- 返回函数参数引用，例如重载 `operator<<` 返回流对象。
- 返回对象内部成员引用，但要保证对象生命周期长于引用使用期。

```cpp
std::ostream& operator<<(std::ostream& os, const X& x) {
    return os;
}
```

> [!TIP]
> 现代 C++ 中不要因为担心拷贝成本而随便返回引用。按值返回通常会有 RVO/NRVO 或移动语义，既安全又高效。

### 28. `extern "C"` 的用法。

#### 面试回答

`extern "C"` 用于告诉 C++ 编译器按 C 语言链接规则处理函数或变量名，避免 C++ 名字改编。它常用于 C/C++ 混合编程、动态库导出 C 接口、让 C 程序调用 C++ 实现。

```cpp
#ifdef __cplusplus
extern "C" {
#endif

int api_init(void);
void api_destroy(void);

#ifdef __cplusplus
}
#endif
```

#### 常见追问

- 不能把重载函数都直接导出为同名 C 接口，因为 C 不支持同名重载。
- `extern "C"` 不改变函数内部实现，C++ 函数体里仍可使用类、模板、异常，但异常不应跨 C ABI 边界传播。
- 头文件要用 `#ifdef __cplusplus`，保证被 C 编译器包含时不会看到 `extern "C"` 语法。

### 29. 内联函数和宏定义的区别。

#### 面试回答

宏是预处理阶段的文本替换，没有类型检查、作用域和调试信息，容易出现优先级和副作用问题。内联函数是正常 C++ 函数，有类型检查、作用域、重载、模板和调试支持。`inline` 不保证一定展开，它主要表示允许函数在多个翻译单元中定义，并给编译器一个内联优化建议。

```cpp
#define SQUARE(x) x * x
int a = SQUARE(1 + 2); // 展开为 1 + 2 * 1 + 2，结果不是 9

inline int square(int x) {
    return x * x;
}
```

#### 补充

现代 C++ 中，常量优先用 `constexpr`/`const`，小函数优先用 `inline` 或模板，宏主要保留给条件编译、头文件保护、平台差异封装等场景。

#### 对比表

| 对比项 | 宏定义 | 内联函数 |
| --- | --- | --- |
| 处理阶段 | 预处理阶段 | 编译阶段 |
| 本质 | 文本替换 | 真正的函数 |
| 类型检查 | 无 | 有 |
| 作用域 | 不遵守 C++ 作用域规则 | 遵守作用域和访问控制 |
| 调试 | 难调试 | 可调试 |
| 重载 | 不支持 | 支持 |
| 副作用 | 参数可能被多次求值 | 参数只求值一次 |

#### 宏的典型问题

```cpp
#define MAX(a, b) ((a) > (b) ? (a) : (b))

int i = 1, j = 2;
int m = MAX(i++, j++); // i 或 j 可能被多次自增
```

内联函数不会有这种多次求值问题：

```cpp
template <class T>
inline const T& maxValue(const T& a, const T& b) {
    return a < b ? b : a;
}
```

#### 常见追问

- **`inline` 一定会展开吗？**  
  不一定。是否展开由编译器根据优化策略决定。

- **为什么头文件里的普通函数常写 `inline`？**  
  因为同一个头文件会被多个 `.cpp` 包含，`inline` 允许函数在多个翻译单元中有相同定义，避免重复定义链接错误。

### 30. C++ 内存分为哪几块？有什么作用？

#### 面试回答

与第 1 题属于同一问题。经典回答是代码区、全局/静态区、常量区、堆区、栈区；更系统的回答是进程虚拟地址空间中的 `.text`、`.rodata`、`.data`、`.bss`、heap、stack、mmap 等区域。

#### 重点补充

- 局部自动变量通常在栈上，不需要手动释放。
- `new/malloc` 申请的对象在堆上，需要 `delete/free` 或 RAII 管理。
- 全局变量和静态变量生命周期贯穿整个程序。
- 字符串字面量通常在只读区，不能修改。

### 31. 静态变量什么时候初始化？

#### 面试回答

具有静态存储期的变量分为静态初始化和动态初始化。零初始化和常量初始化在程序启动早期完成；需要执行代码的动态初始化通常在 `main` 之前完成。函数局部静态变量在第一次执行到定义语句时初始化，C++11 起该初始化是线程安全的。

#### 分类

```cpp
int g;                    // 零初始化，main 前完成
int x = 10;               // 常量初始化，main 前完成
std::string s = "hello";  // 动态初始化，main 前某时刻完成

int f() {
    static int v = compute(); // 第一次调用 f 时初始化
    return v;
}
```

> [!CAUTION]
> 不同翻译单元中的全局对象动态初始化顺序不确定，容易产生“静态初始化顺序问题”。常用函数局部静态对象规避。

### 32. 动态编译与静态编译。

#### 面试回答

面试中这题通常实际想问静态链接和动态链接。静态链接会把用到的库目标代码复制进可执行文件，部署简单但文件大、升级困难；动态链接在运行时加载共享库，多个进程可共享同一份库代码，升级方便但依赖库路径、版本和 ABI。

| 对比 | 静态链接 | 动态链接 |
| --- | --- | --- |
| 产物体积 | 大 | 小 |
| 运行依赖 | 少 | 依赖 `.so/.dll` |
| 升级库 | 需重新链接/发布 | 可替换动态库，需 ABI 兼容 |
| 内存共享 | 差 | 多进程可共享代码页 |
| 启动/加载 | 简单 | 需要动态链接器解析 |

Linux 中可用 `ldd ./app` 查看动态库依赖，用 `-static` 尝试静态链接。

### 33. C++ 和 C 的 `struct` 区别。

#### 面试回答

C 的 `struct` 主要是数据聚合，不能包含成员函数、访问控制和继承。C++ 的 `struct` 本质上接近 `class`，可以有构造函数、析构函数、成员函数、静态成员、继承、虚函数、模板等，只是默认访问权限和默认继承权限是 `public`。

```cpp
struct Point {
    int x{};
    int y{};
    double norm() const {
        return std::sqrt(x * x + y * y);
    }
};
```

在 C 中声明变量常写 `struct Point p;`，若要直接写 `Point p;` 通常需要 `typedef struct Point Point;`。C++ 中类型名可直接使用。

### 34. 拷贝构造函数的调用时机。

#### 面试回答

拷贝构造函数用于用一个同类型对象初始化另一个新对象。常见调用时机包括：对象定义时用已有对象初始化、函数按值传参、函数按值返回、容器扩容或插入时复制元素。现代 C++ 中返回值优化和移动语义会减少真实拷贝，但类型语义上仍要正确支持或显式禁用拷贝。

```cpp
struct A {
    A() = default;
    A(const A&) { std::cout << "copy\n"; }
};

A a;
A b = a;      // 拷贝构造
void f(A x);  // 调用 f(a) 可能拷贝
A g() { return a; } // 可能 RVO/NRVO
```

### 35. `inline` 函数的使用？缺点是什么？

#### 面试回答

`inline` 适合短小、频繁调用、定义在头文件中的函数。它可以减少函数调用开销，也允许函数定义出现在多个翻译单元中而不违反 ODR。缺点是可能增加代码体积、降低指令缓存命中率、暴露实现细节，而且编译器不一定真正内联复杂函数。

#### 适合与不适合

- 适合：简单 getter/setter、小数学函数、模板函数、头文件工具函数。
- 不适合：递归函数、大函数、含复杂循环/异常处理/虚调用的函数。

> [!NOTE]
> 现代优化编译器会自行判断是否内联，`inline` 更多是链接语义和优化建议，不是强制命令。

#### 使用场景

`inline` 适合定义在头文件中的短函数，例如：

```cpp
class Point {
public:
    int x() const { return x_; } // 类内定义，隐式 inline
private:
    int x_{};
};
```

类内定义的成员函数默认具有 inline 语义。模板函数通常也放在头文件中，常与 inline 优化一起出现。

#### 缺点展开

- 代码膨胀：调用点很多时，每处都展开会让可执行文件变大。
- 指令缓存压力：代码变大可能降低 I-cache 命中率。
- 编译依赖增加：函数实现放头文件，修改实现会导致包含它的源文件重新编译。
- ABI 暴露：库头文件暴露更多实现细节，不利于二进制兼容。
- 不适合递归、大循环、复杂控制流函数。

#### 常见追问

- **`inline` 和宏相比优势是什么？**  
  有类型检查、作用域、调试信息，不会重复求值参数。

- **虚函数能不能 inline？**  
  语法上可以。如果编译器能在编译期确定动态类型，也可能内联；但通过基类指针/引用的真正虚调用通常难以内联。

### 36. 为什么拷贝构造函数必须传引用不能传值？

#### 面试回答

如果拷贝构造函数参数按值传递，为了把实参传给形参，需要先拷贝构造这个形参，于是又调用拷贝构造函数，形成无限递归。正确形式是 `T(const T&)`，通常加 `const` 以允许拷贝 const 对象和临时对象。

```cpp
class T {
public:
    T(const T& other); // 正确
    // T(T other);     // 错误思路：传参本身需要拷贝
};
```

如果类支持移动语义，还可以提供 `T(T&&) noexcept`，用于接管右值资源。

### 37. 类中静态函数占用内存么？

#### 面试回答

静态成员函数本身是函数代码，存放在代码段，不占用每个对象的内存。对象大小只与非静态数据成员、虚指针、对齐填充等有关。静态数据成员也不在普通对象内部，而是在全局/静态存储区单独存一份。

```cpp
struct A {
    int x;
    static int count;
    static void f();
};
// sizeof(A) 通常只考虑 x 和对齐，不包含 count 和 f
```

静态成员函数没有 `this`，所以不能直接访问 `x`，除非通过对象或指针访问。

### 38. 构造函数初始化和列表初始化的区别。

#### 面试回答

成员初始化列表是在成员对象构造时直接初始化；构造函数体内赋值是成员先默认构造，再在函数体中赋值。对于 `const` 成员、引用成员、没有默认构造函数的成员对象，必须使用初始化列表。成员真正初始化顺序由类中声明顺序决定，不由初始化列表书写顺序决定。

```cpp
class X {
public:
    X(int v, std::string s) : ref_(v), name_(std::move(s)), value_(v) {}
private:
    int value_;
    int& ref_;
    std::string name_;
};
```

> [!CAUTION]
> 上例实际初始化顺序是 `value_`、`ref_`、`name_`，因为声明顺序如此。初始化列表书写顺序不同可能触发编译器警告。

### 39. 虚函数表的作用和存储的地址。

#### 面试回答

虚函数表用于实现运行期多态。含虚函数的类通常有一张虚表，表中保存虚函数入口地址；对象中通常有一个虚指针 `vptr` 指向所属动态类型的虚表。通过基类指针或引用调用虚函数时，程序根据对象的 `vptr` 找到虚表，再调用对应槽位的函数。

```mermaid
flowchart LR
  Obj[对象: vptr + 成员数据] --> VT[虚函数表]
  VT --> F1[析构函数入口]
  VT --> F2[virtual f()]
  VT --> F3[virtual g()]
```

虚表通常由编译器生成，放在只读数据段或类似区域；`vptr` 是对象的一部分。具体布局由 ABI 决定，C++ 标准没有规定虚表必须如何实现。

#### 1. 虚函数表解决什么问题？

普通成员函数调用在编译期就能确定目标地址，而虚函数调用需要根据对象真实类型决定调用哪个版本。虚函数表就是为运行期查找函数入口服务的。

```cpp
struct Base {
    virtual void f() { std::cout << "Base\n"; }
};

struct Derived : Base {
    void f() override { std::cout << "Derived\n"; }
};

void call(Base& b) {
    b.f(); // 运行时根据 b 实际绑定对象决定调用 Base::f 还是 Derived::f
}
```

在典型实现中，`call` 并不会直接写死 `Base::f` 的地址，而是：

1. 从对象中取出 `vptr`。
2. 通过 `vptr` 找到虚函数表。
3. 根据虚函数在表中的槽位取出函数地址。
4. 间接调用该函数地址。

#### 2. 对象、虚指针、虚表的关系

- 每个含虚函数的类通常有一张或多张虚表。
- 每个含虚函数的对象通常有一个或多个虚指针。
- 同一个类的多个对象共享同一张虚表。
- 对象内只保存 `vptr`，不保存整张虚表。
- 多继承、虚继承下可能存在多个 `vptr` 和更复杂的偏移调整。

#### 3. 虚表一般存放在哪里？

虚表是编译器生成的静态数据，通常放在可执行文件或动态库的只读数据段，如 `.rodata`。对象的 `vptr` 位于对象内存中，一般在对象起始位置，但这不是 C++ 标准保证，只是常见 ABI 实现。

#### 4. 常见追问

- **虚函数调用有什么开销？**  
  多一次间接寻址，可能影响内联和分支预测。但大多数业务场景开销很小，性能敏感内层循环才需要重点评估。

- **构造函数中调用虚函数会动态绑定吗？**  
  不会绑定到尚未构造的派生类部分。构造基类时，当前对象视为基类对象。

- **静态函数能进虚表吗？**  
  不能。静态成员函数没有 `this`，不能作为虚函数。

### 40. 类指针初始化为空指针后调用成员函数会出问题吗？

#### 面试回答

通过空指针调用非静态成员函数是未定义行为。某些情况下函数体没有访问成员变量，看起来可能“能运行”，但这不是合法语义，不能依赖。静态成员函数不需要对象，可以通过类名调用，不存在空对象问题。

```cpp
struct A {
    void f() { std::cout << "f\n"; }
    void g() { std::cout << x << "\n"; }
    static void h() {}
    int x{};
};

A* p = nullptr;
// p->f(); // 未定义行为，即使可能打印
// p->g(); // 大概率崩溃
A::h();    // 正确
```

### 41. 指针和引用的区别？

与第 26 题重复。复习时不要只背“指针可空、引用不可空”，要从语义、生命周期和接口设计三个角度回答。

#### 快速复习版

- 指针是对象，保存地址；引用是已有对象的别名。
- 指针可以不初始化，引用必须初始化。
- 指针可以重新指向其他对象，引用不能重新绑定。
- 指针可以为 `nullptr`，引用语义上必须绑定有效对象。
- 指针可以做算术运算，引用不行。
- `sizeof(pointer)` 得到指针大小，`sizeof(reference)` 得到被引用对象大小。

#### 接口设计版

```cpp
void readOnly(const std::string& s); // 必须传对象，只读
void modify(std::string& s);         // 必须传对象，可修改
void optional(std::string* s);       // 可以不传对象
```

面试时可以补一句：引用并不负责生命周期，返回引用或保存引用时仍要保证被引用对象活得足够久。

### 42. 动态指针的判空。

#### 面试回答

裸指针判空使用 `p == nullptr` 或 `if (p)`。智能指针也提供布尔转换，`if (sp)` 表示当前是否持有对象。释放裸指针后建议置为 `nullptr`，避免悬空指针；但更根本的做法是使用 RAII 和智能指针管理所有权。

```cpp
int* p = new int(1);
if (p != nullptr) {
    std::cout << *p << "\n";
}
delete p;
p = nullptr;

std::unique_ptr<int> up = std::make_unique<int>(2);
if (up) {
    std::cout << *up << "\n";
}
```

> [!CAUTION]
> 判空只能判断指针值是否为空，不能判断它是否悬空。一个非空指针也可能指向已经释放的内存。

### 43. 构造函数可以设置成虚函数吗，为什么？

#### 面试回答

构造函数不能是虚函数。虚函数调用依赖对象的动态类型和虚指针，而构造函数的任务正是创建对象、初始化虚指针和成员；在构造过程中，派生类部分还没有构造完成，不存在完整的动态类型。因此 C++ 不允许虚构造函数。

#### 补充

如果需要“根据条件创建不同派生类对象”，常用工厂模式：

```cpp
struct Base { virtual ~Base() = default; };
struct D1 : Base {};
struct D2 : Base {};

std::unique_ptr<Base> create(int type) {
    if (type == 1) return std::make_unique<D1>();
    return std::make_unique<D2>();
}
```

析构函数可以且经常应该是虚函数，尤其当类作为多态基类使用时。

### 44. `new` 和 `malloc` 有什么区别？`new` 为什么不需要类型转换？

#### 面试回答

`new` 是 C++ 运算符，语法中已经带有目标类型，例如 `new T(args)`，编译器知道要分配 `sizeof(T)` 字节、满足 `T` 的对齐要求，并在分配成功后调用构造函数，所以返回类型天然是 `T*`，不需要类型转换。`malloc` 只接收字节数，返回 `void*`，不会构造对象。

```cpp
auto* p = new std::string("abc"); // 分配 + 构造，返回 std::string*
delete p;                         // 析构 + 释放

void* raw = std::malloc(sizeof(std::string));
auto* q = new (raw) std::string("abc"); // placement new 手动构造
q->~basic_string();
std::free(raw);
```

> [!TIP]
> 第 24 题问总体区别，第 44 题更偏向 `new` 的底层步骤：调用 `operator new` 分配原始内存，再调用构造函数。

#### 为什么 `malloc` 在 C++ 中经常要强转，而 `new` 不需要？

在 C 中，`void*` 可以隐式转换为任意对象指针，因此 `malloc` 返回值不需要强转。在 C++ 中，`void*` 不能隐式转换为 `T*`，所以如果直接用 `malloc`，需要显式转换：

```cpp
int* p = static_cast<int*>(std::malloc(sizeof(int)));
```

而 `new int` 的表达式类型就是 `int*`：

```cpp
int* q = new int(1);
```

这说明 `new` 是类型感知的，编译器从语法上知道目标类型。

#### 底层步骤对比

```cpp
auto* p = new T(args);
```

约等于：

1. 计算 `sizeof(T)` 和对齐。
2. 调用 `operator new` 分配原始内存。
3. 调用 `T::T(args)` 构造对象。
4. 返回 `T*`。

```cpp
void* raw = malloc(sizeof(T));
```

只完成“分配若干字节”，没有对象生命周期。对非平凡类型来说，仅 `malloc` 后把地址当作对象使用是不正确的。

### 45. 虚函数表里存放的内容是什么时候写进去的？

#### 面试回答

虚函数表的内容由编译器在编译/链接阶段生成，里面保存虚函数入口地址、析构函数入口、RTTI 相关信息等实现相关内容。运行时每个对象构造时会设置对象内的虚指针 `vptr`，让它指向当前构造阶段对应类的虚表。不是每创建一个对象就重新生成一张虚表。

#### 构造/析构中的虚表变化

构造基类部分时，`vptr` 指向基类虚表；构造到派生类部分后，`vptr` 再调整为派生类虚表。析构时顺序相反。因此构造函数和析构函数里调用虚函数，不会分派到尚未构造或已经析构的派生类实现。

#### 详细说明

虚表本身不是运行时动态“写入”的普通对象数据，而是编译器根据类的虚函数声明、继承关系和覆盖关系生成的静态表。链接阶段会把这些表和函数符号关联起来。程序运行时创建对象，只是把对象里的 `vptr` 设置为合适的虚表地址。

```cpp
struct Base {
    Base() { f(); }
    virtual void f() { std::cout << "Base\n"; }
    virtual ~Base() = default;
};

struct Derived : Base {
    void f() override { std::cout << "Derived\n"; }
};

Derived d; // Base 构造阶段调用 Base::f，不是 Derived::f
```

可以把过程理解为：

1. 编译器为 `Base` 生成 `Base` 的虚表。
2. 编译器为 `Derived` 生成 `Derived` 的虚表，覆盖槽位指向 `Derived::f`。
3. 构造 `Base` 子对象时，对象 `vptr` 指向 `Base` 虚表。
4. 构造 `Derived` 部分时，`vptr` 改为指向 `Derived` 虚表。

> [!NOTE]
> 原题括号中写“编译阶段”大体方向对，但更严谨地说：虚表内容由编译器生成并经链接确定，运行时构造对象时设置的是对象内的虚指针。

### 46. 虚函数和纯虚函数的区别。

#### 面试回答

虚函数用 `virtual` 声明，有默认实现，派生类可以选择覆盖；纯虚函数在声明末尾写 `= 0`，表示基类只定义接口不提供完整实现，使类成为抽象类，不能直接实例化。派生类必须实现所有纯虚函数后才是具体类。

```cpp
struct Shape {
    virtual ~Shape() = default;
    virtual double area() const = 0; // 纯虚函数
    virtual void print() const {     // 普通虚函数
        std::cout << "shape\n";
    }
};
```

纯虚函数也可以在类外提供定义，纯虚析构函数尤其必须提供定义，因为析构派生对象时会调用基类析构。

#### 虚函数

虚函数用于表达“基类提供默认行为，派生类可以改写”。如果派生类不重写，就继承基类版本。

```cpp
struct Logger {
    virtual void flush() {
        std::cout << "default flush\n";
    }
};
```

特点：

- 类可以实例化，除非还有其他纯虚函数。
- 可以有函数体。
- 派生类可以重写，也可以不重写。
- 通过基类指针/引用调用时支持动态绑定。

#### 纯虚函数

纯虚函数用于表达“基类只定义接口，派生类必须提供实现”。含纯虚函数的类是抽象类。

```cpp
struct Reader {
    virtual std::string read() = 0;
};
```

特点：

- 所在类不能直接实例化。
- 派生类必须实现所有纯虚函数，否则派生类仍是抽象类。
- 可以有定义，但仍然是纯虚函数。
- 常用于接口类、抽象基类、框架扩展点。

#### 纯虚析构函数

纯虚析构函数常用于让类成为抽象类，但它必须提供定义：

```cpp
struct Interface {
    virtual ~Interface() = 0;
};

Interface::~Interface() = default;
```

原因是析构派生类对象时，析构链一定会调用基类析构函数；如果纯虚析构没有定义，会链接失败。

#### 对比表

| 对比项 | 虚函数 | 纯虚函数 |
| --- | --- | --- |
| 声明方式 | `virtual void f();` | `virtual void f() = 0;` |
| 是否必须实现 | 基类通常有实现 | 派生类通常必须实现 |
| 类能否实例化 | 可以 | 含纯虚函数则不能 |
| 设计含义 | 默认行为可覆盖 | 接口约束 |

### 47. 为什么析构函数一般写成虚函数？

#### 面试回答

如果一个类作为多态基类使用，可能通过基类指针删除派生类对象，此时基类析构函数必须是虚函数。否则 `delete basePtr` 只会调用基类析构，派生类资源无法释放，行为未定义。虚析构保证析构按“派生类到基类”的顺序完整执行。

```cpp
struct Base {
    virtual ~Base() = default;
};
struct Derived : Base {
    ~Derived() override { /* release resource */ }
};

Base* p = new Derived;
delete p; // 正确调用 Derived::~Derived 再 Base::~Base
```

不作为基类使用、或不允许通过基类指针删除的类，可以不写虚析构，以节省虚表开销。

#### 为什么非虚析构会出问题？

假设基类析构不是虚函数：

```cpp
struct Base {
    ~Base() { std::cout << "~Base\n"; }
};

struct Derived : Base {
    ~Derived() { std::cout << "~Derived\n"; }
};

Base* p = new Derived;
delete p; // 未定义行为，通常只调用 ~Base
```

如果 `Derived` 中有堆内存、文件句柄、锁、socket 等资源，派生类析构不执行就会泄漏资源。

#### 什么时候必须虚析构？

- 类中有虚函数，作为多态基类使用。
- 可能通过基类指针或引用管理派生对象生命周期。
- 基类提供接口，派生类负责具体资源。

#### 什么时候可以不虚析构？

- 类不作为基类。
- 类只用于静态多态或值语义。
- 基类析构是 `protected` 且不允许外部通过基类指针删除。

```cpp
class NonDeletableBase {
protected:
    ~NonDeletableBase() = default;
};
```

#### 常见追问

- **构造函数为什么不能虚，析构函数为什么可以虚？**  
  构造时对象还没形成完整动态类型；析构时对象曾经是完整动态类型，需要按派生到基类顺序释放资源。

- **有虚析构会增加什么成本？**  
  类通常会有虚表和对象内虚指针，增加一点对象大小和间接调用成本。

### 48. 动态多态和静态多态的实现过程。

#### 面试回答

动态多态通过继承、虚函数、基类指针/引用实现，调用目标在运行期根据对象动态类型确定；静态多态通过函数重载、模板、CRTP 等实现，调用目标在编译期确定。动态多态灵活，适合运行期扩展；静态多态无虚调用开销，适合泛型库和性能敏感代码。

```cpp
// 动态多态
struct Animal { virtual void speak() = 0; };
struct Dog : Animal { void speak() override {} };
void call(Animal& a) { a.speak(); }

// 静态多态
template <class T>
void callStatic(T& x) { x.speak(); }
```

#### 动态多态的实现过程

动态多态依赖虚函数机制，核心是 `vptr + vtable`：

1. 基类声明虚函数。
2. 派生类重写虚函数。
3. 使用基类指针或引用指向派生类对象。
4. 调用虚函数时，通过对象 `vptr` 找到虚表。
5. 从虚表中取出实际函数入口并调用。

优点：

- 运行期根据实际对象类型选择行为。
- 接口稳定，扩展派生类时调用端不需要改。
- 适合插件、框架、业务策略对象等场景。

缺点：

- 有一次间接调用开销。
- 可能阻碍内联。
- 需要通过指针/引用管理对象生命周期。

#### 静态多态的实现过程

静态多态在编译期完成函数选择或代码生成，常见方式有函数重载、运算符重载、模板、CRTP。

```cpp
template <class Derived>
struct BaseT {
    void interface() {
        static_cast<Derived*>(this)->impl();
    }
};

struct Impl : BaseT<Impl> {
    void impl() {
        std::cout << "Impl\n";
    }
};
```

优点：

- 编译期确定调用目标，容易内联。
- 无虚表、虚指针开销。
- 类型错误更早暴露。

缺点：

- 模板代码可能增加编译时间和代码体积。
- 接口变更会影响编译依赖。
- 不适合运行期才知道具体类型的场景。

#### 对比表

| 对比项 | 动态多态 | 静态多态 |
| --- | --- | --- |
| 实现方式 | 继承 + 虚函数 | 重载、模板、CRTP |
| 绑定时机 | 运行期 | 编译期 |
| 开销 | 虚调用、可能有 vptr | 通常零运行期开销 |
| 扩展方式 | 新增派生类 | 新增类型/模板实例 |
| 适合场景 | 运行期对象异构集合 | 泛型算法、性能敏感代码 |

### 49. STL 中 `vector` 删除元素，迭代器如何变化？为什么常见两倍扩容？如何释放空间？

#### 面试回答

`vector::erase` 删除元素后，会使删除位置及其后的迭代器、指针、引用失效，并返回删除位置后一个有效迭代器。`vector` 扩容通常按倍数增长，是为了让尾插达到均摊 O(1)，同时控制空间浪费。释放容量可用 `shrink_to_fit()` 请求收缩，或用临时对象 `swap` 强制释放。

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (*it % 2 == 0) it = v.erase(it);
    else ++it;
}

std::vector<int>().swap(v); // 清空并释放容量
```

#### 扩容说明

每次扩容都会重新分配连续内存，并移动/拷贝旧元素，所以所有迭代器失效。增长因子常见为 1.5 或 2，标准不规定必须两倍。倍增能避免每次 `push_back` 都重新分配，降低总搬迁次数。

#### 1. 删除元素后的迭代器变化

`vector` 是连续数组，删除中间元素后，后面的元素会向前移动填补空位。因此：

- 删除位置之前的迭代器和引用仍然有效。
- 删除位置及其之后的迭代器、指针、引用全部失效。
- `erase` 返回删除位置后新的有效迭代器。
- 如果删除的是最后一个元素，返回新的 `end()`。

```cpp
std::vector<int> v{1, 2, 3, 4};
auto it = v.begin() + 1; // 指向 2
it = v.erase(it);        // 删除 2，it 指向 3
```

边遍历边删除必须使用返回值：

```cpp
for (auto it = v.begin(); it != v.end(); ) {
    if (needRemove(*it)) {
        it = v.erase(it);
    } else {
        ++it;
    }
}
```

#### 2. 为什么扩容常用倍增？

如果每次 `push_back` 都只增加 1 个容量，那么插入 n 个元素可能反复重新分配和搬迁，复杂度接近 O(n^2)。倍增扩容可以保证总搬迁次数受控，使尾插的均摊复杂度为 O(1)。

简单理解：

- 容量从 1、2、4、8、16 增长时，插入 n 个元素最多搬迁约 `1 + 2 + 4 + ... < 2n` 次。
- 平均到每次插入，搬迁成本仍是常数级。

标准没有规定必须 2 倍，不同 STL 实现可能使用 1.5 倍或 2 倍。增长因子越大，扩容次数越少但空间浪费越多；增长因子越小，空间更省但搬迁更频繁。

#### 3. 如何释放空间？

`clear()` 只销毁元素，让 `size()` 变为 0，但通常保留 `capacity()`。释放容量可以：

```cpp
v.clear();          // 元素没了，容量通常还在
v.shrink_to_fit();  // 请求收缩，标准不强制保证
```

强制释放常用 swap 技巧：

```cpp
std::vector<int>().swap(v);
```

如果只是删除一部分元素但还要继续使用，未必应该释放容量，因为后续再插入还会重新分配。

#### 常见追问

- **`reserve` 和 `resize` 区别？**  
  `reserve` 改容量，不创建元素；`resize` 改大小，会创建或销毁元素。

- **哪些操作会导致 `vector` 迭代器全部失效？**  
  触发重新分配的 `push_back/emplace_back/insert/reserve` 等操作会让所有迭代器、指针、引用失效。

- **为什么 `vector` 随机访问快？**  
  因为元素连续存储，可通过首地址加偏移直接定位，缓存局部性也好。

### 50. `map`、`set` 如何实现？为什么使用红黑树？

#### 面试回答

`std::map` 和 `std::set` 通常基于红黑树实现。`map` 存 key-value，按 key 排序；`set` 只存 key，本质上 key 也是 value。红黑树是近似平衡二叉搜索树，能保证插入、删除、查找都是 O(log n)，并且中序遍历天然有序。

#### 为什么不用普通 BST 或 AVL

- 普通 BST 在有序插入时可能退化成链表，复杂度 O(n)。
- AVL 更严格平衡，查找略快，但插入删除可能旋转更多。
- 红黑树平衡要求相对宽松，插入删除调整成本低，综合性能适合标准库关联容器。

`map` 和 `set` 可以共用同一套树结构，只是节点保存的数据和比较方式不同。

#### 1. `map` 和 `set` 的数据模型

`set` 保存的是 key 集合，key 不能重复，遍历结果按 key 有序。`map` 保存的是 key-value 映射，key 不能重复，按 key 排序，value 不参与排序。

```cpp
std::set<int> s;             // 节点保存 int
std::map<int, std::string> m; // 节点保存 pair<const int, string>
```

在 `map` 中，key 通常是 `const`，不能通过迭代器直接修改 key，因为修改 key 会破坏树的有序性。若要改 key，通常删除旧节点再插入新节点。

#### 2. 红黑树的关键性质

红黑树是二叉搜索树，并额外维护颜色约束：

- 每个节点是红色或黑色。
- 根节点是黑色。
- 所有叶子 NIL 节点视为黑色。
- 红色节点不能有红色子节点。
- 从任一节点到其后代 NIL 节点的所有路径包含相同数量的黑色节点。

这些性质保证最长路径不会超过最短路径的两倍，因此树高是 O(log n)，查找、插入、删除都能保持 O(log n)。

#### 3. 为什么适合标准库关联容器？

标准库的有序关联容器需要：

- 按 key 有序遍历。
- 支持范围查询，如 `lower_bound/upper_bound`。
- 插入删除后仍保持有序。
- 查找、插入、删除复杂度稳定。

红黑树在查找性能和插删调整成本之间折中较好。AVL 树更平衡，但插删旋转可能更多；哈希表平均查找更快，但无序且不适合范围查询。

#### 4. 常见追问

- **`map` 和 `unordered_map` 区别？**  
  `map` 基于有序树，O(log n)，支持有序遍历和范围查询；`unordered_map` 基于哈希表，平均 O(1)，但无序，最坏可能 O(n)。

- **红黑树为什么查找是 O(log n)？**  
  因为颜色约束限制树高为 O(log n)，二叉搜索树查找最多沿树高走一条路径。

- **`map` 和 `set` 为什么能用同一棵红黑树实现？**  
  树只需要按 key 排序和比较；`set` 的 value 就是 key，`map` 的 value 是 pair，但排序只看 pair 中的 key。

### 51. 模板和实现可不可以不写在一个文件里面？为什么？

#### 面试回答

普通模板的定义通常要放在头文件中，因为模板不是直接编译成某个固定函数，而是在使用具体类型时由编译器实例化。实例化时编译器必须看到完整定义，否则无法生成代码。也可以把实现放在 `.tpp/.ipp` 后由头文件包含，或者使用显式实例化把少数固定类型的实现放在 `.cpp`。

```cpp
// foo.h
template <class T>
T add(T a, T b) {
    return a + b;
}
```

显式实例化方式：

```cpp
// foo.cpp
template int add<int>(int, int);
template double add<double>(double, double);
```

这种方式只能支持显式列出的类型，不适合任意泛型使用。

### 52. 请简述 C++11 的新特性。

#### 面试回答

C++11 是现代 C++ 的基础，引入了类型推导、移动语义、智能指针、lambda、并发库、`nullptr`、范围 for、统一初始化、`constexpr`、强类型枚举、可变参数模板等。面试中重点要能讲清右值引用/移动语义、智能指针、lambda、`auto`、`nullptr` 和线程库。

#### 高频特性

| 特性 | 作用 |
| --- | --- |
| `auto/decltype` | 类型推导，减少冗长类型 |
| 右值引用/移动语义 | 避免不必要深拷贝，转移资源 |
| `unique_ptr/shared_ptr/weak_ptr` | RAII 管理动态资源 |
| lambda | 本地匿名函数，适合回调和算法 |
| `nullptr` | 类型安全空指针 |
| 范围 for | 简化容器遍历 |
| `thread/mutex/condition_variable` | 标准并发编程 |
| `constexpr` | 编译期常量计算 |
| `enum class` | 强类型枚举，避免隐式转换 |

```cpp
auto p = std::make_unique<int>(42);
std::vector<int> v{1, 2, 3};
std::for_each(v.begin(), v.end(), [](int x) { std::cout << x; });
```

#### 1. 类型推导：`auto` 和 `decltype`

`auto` 让编译器根据初始化表达式推导变量类型，适合迭代器、lambda 返回值等复杂类型。`decltype` 根据表达式推导类型，常用于模板。

```cpp
auto it = v.begin();
decltype(v.size()) n = v.size();
```

注意 `auto` 默认会丢掉顶层 `const` 和引用，需要时写 `const auto&`。

#### 2. 右值引用和移动语义

右值引用 `T&&` 允许绑定即将销毁的临时对象，从而“窃取”资源，避免深拷贝。移动语义对 `std::vector`、`std::string`、智能指针等资源管理类非常重要。

```cpp
std::string a = "hello";
std::string b = std::move(a); // b 接管资源，a 仍有效但值未指定
```

#### 3. 智能指针

C++11 标准化了 `unique_ptr/shared_ptr/weak_ptr`，用 RAII 管理动态资源，减少手写 `new/delete`。

```cpp
auto p = std::make_shared<int>(1);
```

#### 4. lambda 表达式

lambda 提供局部匿名函数对象，常用于 STL 算法、回调和异步任务。

```cpp
std::sort(v.begin(), v.end(), [](int a, int b) {
    return a > b;
});
```

#### 5. `nullptr`

`nullptr` 是类型安全的空指针，类型是 `std::nullptr_t`，解决 `NULL` 可能是整数 0 导致的重载歧义。

```cpp
void f(int);
void f(char*);

f(nullptr); // 调用 f(char*)
```

#### 6. 并发库

C++11 引入 `std::thread`、`std::mutex`、`std::condition_variable`、`std::future`、`std::atomic`，使跨平台并发编程进入标准库。

#### 7. 统一初始化和 initializer_list

花括号初始化统一了对象、数组、容器的初始化写法，并能减少窄化转换。

```cpp
std::vector<int> nums{1, 2, 3};
int x{10};
```

#### 8. 其它常见特性

- `enum class`：强类型枚举，避免隐式转换和命名污染。
- `constexpr`：编译期常量计算。
- `default/delete`：显式默认或删除特殊成员函数。
- 可变参数模板：支持类型安全的变长参数。
- 委托构造：一个构造函数调用另一个构造函数。
- `override/final`：增强虚函数覆盖检查和继承控制。

#### 常见追问

- **C++11 最核心的变化是什么？**  
  从工程角度看，是移动语义、RAII 智能指针、lambda 和标准并发库，它们明显改变了现代 C++ 写法。

- **`move` 是否真的移动？**  
  `std::move` 本身只是强制转换为右值引用，真正移动发生在移动构造或移动赋值函数中。

### 53. 说一说 lambda 函数。

#### 面试回答

lambda 是 C++ 中可在局部定义的匿名函数对象，语法是 `[capture](params) -> ret { body }`。捕获列表决定 lambda 如何访问外部变量，可按值、按引用或显式捕获。lambda 常用于 STL 算法、回调、异步任务、定制比较器等。

```cpp
int base = 10;
auto f = [base](int x) { return x + base; }; // 按值捕获
auto g = [&base](int x) { base += x; };      // 按引用捕获
```

#### 捕获方式

- `[=]`：默认按值捕获用到的外部变量。
- `[&]`：默认按引用捕获。
- `[x, &y]`：显式捕获 x 的副本和 y 的引用。
- `[this]`：捕获当前对象指针。
- C++14 支持初始化捕获，如 `[p = std::move(ptr)]`。

> [!CAUTION]
> 引用捕获要特别注意生命周期。如果 lambda 在异步线程或回调中执行，引用的局部变量可能已经销毁，导致悬空引用。

#### lambda 的本质

lambda 表达式会生成一个匿名闭包类，捕获的变量会成为这个闭包对象的成员，函数体对应闭包类的 `operator()`。

```cpp
int x = 1;
auto f = [x](int y) { return x + y; };
```

可粗略理解为编译器生成：

```cpp
class Closure {
public:
    explicit Closure(int x) : x_(x) {}
    int operator()(int y) const { return x_ + y; }
private:
    int x_;
};
```

#### `mutable`

按值捕获的变量在默认情况下不能在 lambda 体内修改，因为 `operator()` 默认是 `const`。如果要修改捕获副本，需要加 `mutable`。

```cpp
int x = 1;
auto f = [x]() mutable {
    ++x;       // 修改的是副本
    return x;
};
```

#### 返回类型

简单 lambda 返回类型可由编译器推导；复杂分支或想显式说明时可写 `-> ret`。

```cpp
auto div = [](int a, int b) -> double {
    return static_cast<double>(a) / b;
};
```

#### 常见使用场景

- STL 算法比较器：`sort/find_if/count_if`。
- 回调函数。
- 线程任务。
- 延迟执行逻辑。
- 局部封装一小段只在当前函数使用的逻辑。

#### 常见追问

- **按值捕获是否一定安全？**  
  对普通变量通常更安全，但如果捕获的是指针，指针指向对象的生命周期仍要自己保证。

- **`[this]` 捕获有什么风险？**  
  捕获的是 `this` 指针。如果对象销毁后 lambda 仍执行，会悬空。C++17 可用 `[*this]` 捕获对象副本，但要注意拷贝成本和语义。

- **无捕获 lambda 能否转函数指针？**  
  可以。无捕获 lambda 可隐式转换为对应函数指针；有捕获 lambda 不行，因为它需要保存状态。

### 54. C++ 中三种智能指针解决的问题以及区别？

与第 20 题相关，但可以按“解决什么问题”记忆：

- `unique_ptr` 解决独占资源自动释放问题，表达唯一所有权，适合类成员、工厂返回值。
- `shared_ptr` 解决多个对象共享同一资源生命周期的问题，通过控制块保存强引用计数、弱引用计数和 deleter。
- `weak_ptr` 解决 `shared_ptr` 循环引用和非拥有观察问题，不增加强引用计数，访问前用 `lock()`。

```cpp
struct Node {
    std::shared_ptr<Node> next;
    std::weak_ptr<Node> prev; // 避免双向链表循环引用
};
```

#### 对比表

| 对比项 | `unique_ptr` | `shared_ptr` | `weak_ptr` |
| --- | --- | --- | --- |
| 是否拥有对象 | 拥有 | 拥有 | 不拥有 |
| 所有权模型 | 独占 | 共享 | 弱观察 |
| 是否可拷贝 | 不可拷贝 | 可拷贝 | 可拷贝 |
| 是否可移动 | 可移动 | 可移动 | 可移动 |
| 释放时机 | `unique_ptr` 销毁时 | 最后一个强引用销毁时 | 不控制释放 |
| 典型问题 | 所有权转移后原指针为空 | 循环引用、控制块开销 | 使用前必须 `lock` |

#### 选择原则

优先级通常是：

1. 能用局部对象或容器，就不用动态分配。
2. 需要动态分配且所有权唯一，用 `unique_ptr`。
3. 生命周期确实共享且无法明确唯一拥有者，用 `shared_ptr`。
4. 只观察 `shared_ptr` 管理的对象，用 `weak_ptr`。

> [!TIP]
> 面试中可以强调：智能指针不是为了“把所有裸指针替换掉”，而是为了把所有权表达清楚。非拥有关系仍可用裸指针或引用。

### 55. `shared_ptr` 使用注意事项？

#### 面试回答

`shared_ptr` 使用时要避免重复控制块、循环引用、从裸 `this` 构造、误以为对象本身线程安全等问题。优先使用 `make_shared`，它通常一次分配对象和控制块，效率更高且异常安全更好。

#### 常见坑

```cpp
int* raw = new int(1);
std::shared_ptr<int> p1(raw);
// std::shared_ptr<int> p2(raw); // 错误：两个控制块，最终 double delete
```

- 不要把同一个裸指针交给多个 `shared_ptr` 构造。
- 需要在成员函数中获取管理自己的 `shared_ptr`，使用 `std::enable_shared_from_this`。
- 强引用环会导致引用计数永远不归零，用 `weak_ptr` 打破。
- 引用计数增减是线程安全的，但被管理对象的读写不是自动线程安全。
- `make_shared` 的对象和控制块同一次分配，若还有 `weak_ptr` 存在，对象析构后控制块内存可能延迟释放；特殊大对象场景可权衡。

#### `enable_shared_from_this`

如果对象已经由 `shared_ptr` 管理，成员函数中需要获得指向自己的 `shared_ptr`，不能写 `std::shared_ptr<T>(this)`，否则会产生新的控制块。

正确方式：

```cpp
class Session : public std::enable_shared_from_this<Session> {
public:
    void start() {
        auto self = shared_from_this();
        // 把 self 放入异步回调，延长对象生命周期
    }
};
```

使用前提是对象已经被某个 `shared_ptr` 管理，否则 `shared_from_this()` 会抛异常或出错。

#### 线程安全边界

```cpp
auto sp = std::make_shared<int>(0);
```

多个线程拷贝、销毁不同的 `shared_ptr` 实例，引用计数本身是安全的。但多个线程同时修改 `*sp` 指向的 `int`，仍然是数据竞争，需要锁或原子变量。

#### 常见追问

- **`shared_ptr` 控制块里有什么？**  
  通常有强引用计数、弱引用计数、删除器、分配器，有些实现还会存对象本身。

- **`make_shared` 有什么缺点？**  
  对象和控制块一起分配，如果对象很大且还有 `weak_ptr` 长时间存在，对象内存所在的大块分配可能要等控制块释放后才归还。

### 56. `weak_ptr` 如何解决 `shared_ptr` 的循环引用问题？

#### 面试回答

`shared_ptr` 通过强引用计数决定对象生命周期。如果两个对象互相持有 `shared_ptr`，即使外部引用都释放，二者的强引用计数仍大于 0，析构不会发生。把其中一个方向改成 `weak_ptr`，它不增加强引用计数，就能让对象在外部强引用释放后正常析构。

```cpp
struct Parent;
struct Child {
    std::weak_ptr<Parent> parent;
};
struct Parent {
    std::shared_ptr<Child> child;
};

if (auto p = child->parent.lock()) {
    // parent 仍然存在，可以安全访问
}
```

`weak_ptr::lock()` 会在对象还活着时返回 `shared_ptr`，否则返回空 `shared_ptr`。

#### 循环引用示例

错误写法：

```cpp
struct A;
struct B;

struct A {
    std::shared_ptr<B> b;
    ~A() { std::cout << "~A\n"; }
};

struct B {
    std::shared_ptr<A> a;
    ~B() { std::cout << "~B\n"; }
};

auto a = std::make_shared<A>();
auto b = std::make_shared<B>();
a->b = b;
b->a = a;
// 离开作用域后，A 和 B 的引用计数互相撑住，析构不执行
```

修正写法：

```cpp
struct B {
    std::weak_ptr<A> a;
};
```

#### 为什么 `weak_ptr` 有效？

`weak_ptr` 只增加弱引用计数，不增加强引用计数。对象是否销毁只由强引用计数决定。当外部 `shared_ptr` 都释放后，强引用计数归零，对象析构；控制块会等弱引用也释放后再销毁。

#### 常见场景

- 双向链表中后向指针。
- 父子对象中子对象指回父对象。
- 观察者列表。
- 缓存系统中保存“如果还活着就取用”的对象。

### 57. 假设有一个指针，如何做到多次使用，一次释放？

#### 面试回答

核心是明确所有权。一个资源只能有一个释放责任方，其他地方只是借用。独占所有权用 `unique_ptr`，共享所有权用 `shared_ptr`，非拥有使用裸指针、引用或 `weak_ptr`。不要让多个模块都认为自己负责 `delete` 同一块内存。

```cpp
auto owner = std::make_unique<int>(10);
int* view1 = owner.get(); // 借用，不释放
int& view2 = *owner;      // 借用，不释放
// owner 生命周期结束时自动释放一次
```

如果必须在 C 接口中传递资源，可配合自定义 deleter：

```cpp
using FilePtr = std::unique_ptr<FILE, decltype(&std::fclose)>;
FilePtr fp(std::fopen("a.txt", "r"), &std::fclose);
```

### 58. 在 `if` 条件中如何判断 `bool`、`int`、`float`、`char*` 是否为“零”？

#### 面试回答

`bool` 直接用 `if (!b)`；`int` 用 `i == 0`；浮点数不要直接比较等于 0，应使用误差范围；指针用 `p == nullptr`。

```cpp
bool b = false;
int i = 0;
float f = 0.0f;
char* p = nullptr;

if (!b) {}
if (i == 0) {}
if (std::fabs(f) < 1e-6f) {}
if (p == nullptr) {}
```

> [!NOTE]
> 面试中常考的是代码风格和类型语义：布尔不要写 `b == false` 也可以，但整数和指针最好显式比较，浮点数要考虑精度。

### 59. 以 `int` 变量为对象，设计一个类，给出所有成员函数声明。

#### 面试回答

如果只是封装一个 `int`，可以展示构造、析构、拷贝控制、移动控制、访问器、修改器和常见运算符。由于 `int` 自身不管理资源，实际工程可以遵循 Rule of Zero，让编译器生成特殊成员函数；面试题通常希望你列出完整成员函数。

```cpp
class IntBox {
public:
    IntBox();

    IntBox(IntBox&& other) noexcept;
    ~IntBox();

    IntBox& operator=(const IntBox& other);
    IntBox& operator=(IntBox&& other) noexcept;

    int value() const;

    IntBox& operator++();
```

### 60. 32 位/64 位系统具体指的是什么长度，对系统有何影响？

#### 面试回答

32 位/64 位通常指 CPU 通用寄存器宽度、指令集和进程虚拟地址能力，也会影响 ABI 和指针大小。32 位进程指针通常 4 字节，理论虚拟地址空间 4GB；64 位进程指针通常 8 字节，可寻址空间远大于实际物理内存。

#### 影响

- 指针、`size_t`、部分 long 类型大小变化，结构体对齐和大小变化。
- 64 位程序可使用更大虚拟地址空间，适合大内存应用。
- 指针变大可能增加内存占用和缓存压力。
- 系统调用约定、函数调用 ABI、库文件架构不同。
- 32 位程序在 64 位系统上运行需要兼容库支持。

> [!CAUTION]
> “64 位系统”不表示所有 C++ 类型都是 64 位。比如多数平台 `int` 仍是 32 位。

#### 常见数据模型

不同平台的 C/C++ 基本类型大小模型不同：

| 模型 | int | long | pointer | 常见平台 |
| --- | ---: | ---: | ---: | --- |
| ILP32 | 32 | 32 | 32 | 32 位 Linux/Windows |
| LP64 | 32 | 64 | 64 | 64 位 Linux/macOS |
| LLP64 | 32 | 32 | 64 | 64 位 Windows |

因此不能简单说“64 位下 long 一定 64 位”，在 Windows 64 位中 `long` 仍通常是 32 位。

#### 对程序的具体影响

- 指针字段多的结构体在 64 位下会变大。
- `sizeof(size_t)` 变大，容器可表示更大大小。
- 二进制文件、网络协议、磁盘结构不能直接依赖 `long` 和指针大小。
- C/C++ 与汇编、系统调用、第三方库交互时必须 ABI 一致。

#### 常见追问

- **32 位进程在 64 位系统上指针多大？**  
  看进程本身的位数。32 位进程指针仍通常是 4 字节。

- **64 位程序一定更快吗？**  
  不一定。它能使用更多寄存器和地址空间，但指针变大也可能增加缓存压力。

### 61. 系统物理内存与虚拟内存之间的联系与区别。

#### 面试回答

物理内存是真实 RAM，虚拟内存是操作系统为每个进程提供的地址空间抽象。进程访问的是虚拟地址，CPU 的 MMU 根据页表把虚拟地址转换为物理地址。虚拟内存提供进程隔离、权限保护、按需加载、共享映射、换页和大地址空间等能力。

#### 详细解释

- 每个进程看到的地址从低到高似乎连续，但背后可能映射到不同物理页。
- 多个进程可把同一动态库代码页映射到各自虚拟地址空间，实现共享。
- 访问未映射页会触发缺页异常，内核按需从磁盘加载或分配物理页。
- 物理内存不够时，操作系统可把不活跃页面换出到磁盘，但性能会明显下降。

```text
进程虚拟地址 --> 页表/MMU --> 物理内存页
```

#### 为什么需要虚拟内存？

- **隔离性**：每个进程有独立地址空间，一个进程不能随意访问另一个进程内存。
- **安全性**：页表项带权限位，可控制可读、可写、可执行。
- **便利性**：进程看到连续地址，简化程序和链接器设计。
- **共享**：多个进程可共享同一动态库代码页。
- **按需加载**：程序启动时不必把所有代码和数据都读入物理内存。
- **扩展性**：可通过换页把不常用页面放到磁盘。

#### 地址转换过程

CPU 执行访存指令时产生虚拟地址，MMU 查询页表得到物理页框号，再加页内偏移得到物理地址。为了减少页表查询开销，CPU 有 TLB 缓存近期虚拟页到物理页的映射。

#### 常见追问

- **虚拟内存是不是磁盘？**  
  不是。虚拟内存是地址空间抽象，部分虚拟页可能映射到物理内存，部分可能暂时在磁盘交换区或文件中。

- **缺页异常一定是错误吗？**  
  不一定。按需分配和文件映射首次访问都会触发正常缺页；非法地址访问导致的缺页才会转成段错误。

### 62. 编译器优化级别和基本思想。

#### 面试回答

GCC/Clang 常见优化级别有 `-O0`、`-O1`、`-O2`、`-O3`、`-Os`、`-Og`、`-Ofast`。`-O0` 便于调试；`-O2` 是常用发布优化；`-O3` 更激进，可能增加代码体积；`-Og` 兼顾调试和部分优化；`-Ofast` 可能牺牲标准严格语义，尤其是浮点。

#### 基本优化思想

- 常量折叠、常量传播。
- 死代码删除。
- 函数内联。
- 循环展开、循环不变量外提、向量化。
- 公共子表达式消除。
- 寄存器分配。
- 分支预测和代码布局。
- 链接时优化 LTO、跨函数优化。

> [!CAUTION]
> 优化后调试栈和变量可能不准确，未定义行为在高优化下更容易暴露出“诡异”结果。

#### 常见优化级别

| 级别 | 特点 | 适用场景 |
| --- | --- | --- |
| `-O0` | 基本不优化，调试体验最好 | Debug |
| `-Og` | 保留较好调试体验，同时做部分优化 | 调试优化相关问题 |
| `-O1` | 基础优化 | 一般较少单独使用 |
| `-O2` | 常用优化，不太激进 | Release 默认选择 |
| `-O3` | 更激进，如更多内联、向量化、循环优化 | 性能敏感且验证充分 |
| `-Os/-Oz` | 优化代码体积 | 嵌入式、体积敏感 |
| `-Ofast` | 允许违反部分标准语义换性能 | 数值代码需谨慎 |

#### 优化示例

```cpp
int f() {
    int x = 1 + 2; // 常量折叠为 3
    return x;
}
```

编译器可能直接生成返回 3 的代码。

#### 常见追问

- **为什么 Release 出现 Debug 没有的问题？**  
  常见原因是未定义行为、未初始化变量、越界、违反严格别名规则、数据竞争。优化只是让问题暴露得更明显。

- **如何让性能分析更准？**  
  用 Release 优化级别编译，保留符号 `-g`，必要时加 `-fno-omit-frame-pointer` 方便采样栈回溯。

### 63. bool less(float x, float y) { return *(int *)&x < *(int *)&y; } 是否正确？对应哪种类型转换？

#### 面试回答

不正确。这个写法把 `float*` 强行当作 `int*` 解引用，违反严格别名规则，也可能有对齐问题。即使暂时能运行，把 IEEE 754 浮点位模式当作有符号整数比较，也不能正确处理负数、`-0`、NaN 等情况。它属于 C 风格强制转换，接近 `reinterpret_cast<int*>(&x)` 的危险用法。

```cpp
bool less(float x, float y) {
    return x < y; // 正确
}
```

如果确实要查看浮点位模式，C++20 可用 `std::bit_cast`，但查看位模式不等于能按整数比较大小。

#### 为什么按整数比较浮点位模式不可靠？

IEEE 754 浮点数的编码不是简单的有符号整数顺序。正数部分在某些条件下位模式和数值大小有相关性，但一旦涉及负数、正负零、NaN、无穷大，就会出错。

例如：

- `-1.0` 的最高位为 1，如果当作有符号整数比较，排序关系会被破坏。
- `+0.0` 和 `-0.0` 数值相等，但位模式不同。
- NaN 和任何数比较都为 false，包括和自身比较。

#### 类型系统问题

`*(int*)&x` 同时涉及：

- C 风格强转，意图不清晰。
- 违反严格别名规则：用 `int*` 访问 `float` 对象。
- 可能违反对齐要求。

如果只是想复制位模式：

```cpp
float x = 1.0f;
auto bits = std::bit_cast<std::uint32_t>(x); // C++20，安全表达“查看位模式”
```

但比较大小仍应使用浮点比较本身。

### 64. `char* p = "abcdefg...z"` 强转成 `int*` 再自增，求该值。

#### 面试回答

这题原题不严谨。现代 C++ 中字符串字面量类型是 `const char[N]`，不应赋给 `char*`。把字符指针强转为 `int*` 后自增，指针会按 `sizeof(int)` 前进，常见平台是前进 4 字节；但若解引用读取值，会受到对齐、严格别名、端序和平台影响，不是可移植行为。

```cpp
const char* p = "abcdefghijklmnopqrstuvwxyz";
auto ip = reinterpret_cast<const int*>(p);
++ip; // 地址通常增加 sizeof(int)，例如 4 字节
```

> [!CAUTION]
> 只能回答“指针地址偏移多少”相对明确；如果问解引用得到什么整数，必须说明小端/大端、对齐和未定义行为风险。

#### 分情况说明

如果只做指针自增：

```cpp
const char* p = "abcdefghijklmnopqrstuvwxyz";
auto ip = reinterpret_cast<const int*>(p);
++ip;
```

`ip` 的地址会从原地址增加 `sizeof(int)` 字节。多数常见平台 `sizeof(int) == 4`，所以会跳过 `'a' 'b' 'c' 'd'`，指向从 `'e'` 开始的位置对应的地址。

如果继续解引用：

```cpp
int value = *ip;
```

就存在多重问题：

- 地址可能不满足 `int` 对齐要求。
- 用 `int*` 访问字符数组违反类型别名规则。
- 读取结果依赖大小端。
- 字符串字面量位于只读存储区，不应通过非 const 指针修改。

#### 面试回答建议

这类题不要直接给一个神秘数字。更好的回答是：

1. 先修正 `char*` 为 `const char*`。
2. 说明 `int*` 自增按 `sizeof(int)` 前进，通常是 4 字节。
3. 再说明如果解引用则不具备可移植性，甚至可能是未定义行为。

### 65. 谈一下模板 `template`。

#### 面试回答

模板是 C++ 的泛型编程机制，可以把类型或值作为参数，在编译期生成具体代码。常见有函数模板、类模板、变量模板、别名模板。模板让 STL 容器和算法可以适配多种类型，同时保持类型安全和零运行期开销。

```cpp
template <class T>
T maxValue(const T& a, const T& b) {
    return a < b ? b : a;
}

template <class T>
class Box {
public:
    explicit Box(T value) : value_(std::move(value)) {}
private:
    T value_;
};
```

#### 进阶点

- 模板实例化发生在编译期，编译器要看到定义。
- 支持特化和偏特化，为特定类型定制实现。
- 可变参数模板支持任意数量参数。
- 模板元编程可在编译期计算，但会增加编译复杂度。
- C++20 concepts 可约束模板参数，错误信息更友好。

### 66. 定义函数时，缺省函数的返回值类型，则函数的返回值类型为？

> [!CAUTION]
> 该题属于旧 C 语言语境。早期 C 有“未写返回类型默认为 int”的历史规则，但现代 C 已不推荐/不支持这种写法，C++ 普通函数从来不能省略返回类型。

#### 修正后的问题

在现代 C/C++ 中，函数定义时是否可以省略返回值类型？

#### 答案

现代 C++ 中不可以省略普通函数返回类型。C++14 起可以写 `auto f() { return 1; }` 让编译器推导返回类型，但 `auto` 本身仍然是显式返回类型占位符，并不是“缺省”。如果是旧式选择题，按历史 C 语境可能选 `int`，但工程面试应回答：现代 C++ 中非法，必须显式声明。

```cpp
// f() { return 1; }       // C++ 非法
auto f() { return 1; }     // C++14 起可返回类型推导
int g() { return 1; }      // 明确返回 int
```

### 67. 要实现动态联编，必须使用什么调用虚函数？

#### 面试回答

动态联编也叫动态绑定，必须通过基类指针或基类引用调用虚函数，才能根据对象的实际动态类型在运行期分派到派生类实现。如果直接通过对象名调用，或者发生对象切片，通常会按静态类型绑定，达不到多态效果。原选择题中最合适答案是“基类指针”，但基类引用同样可以实现动态绑定。

```cpp
struct Base {
    virtual void f() { std::cout << "Base\n"; }
};
struct Derived : Base {
    void f() override { std::cout << "Derived\n"; }
};

Derived d;
Base* p = &d;
Base& r = d;
p->f(); // Derived
r.f();  // Derived

Base b = d; // 对象切片
b.f();      // Base
```
